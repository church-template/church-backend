## Major

### `docs/superpowers/specs/2026-06-04-g2-common-exception-design.md:36` — `media`/`gallery_photos`가 `BaseTimeEntity`를 상속하면 스펙에 없는 `updated_at` 컬럼이 강제됨

문서의 매트릭스는 `media / gallery_photos`의 `updated_at`을 명시적으로 `✗`로 표시하지만, 같은 행의 상속은 `BaseTimeEntity`입니다. 그런데 `BaseTimeEntity` 산출물 정의는 `createdAt`과 `updatedAt`을 모두 포함합니다.

영향: 이후 도메인 마이그레이션이 원본 스펙대로 `created_at`만 만들면 `ddl-auto=validate`에서 실패하고, 반대로 마이그레이션에 `updated_at`을 추가하면 불필요 컬럼을 줄이겠다는 D1 근거와 원본 테이블 스펙이 깨집니다.

제안: `media`와 `gallery_photos`는 공통 상속 대상에서 빼고 각 엔티티가 `createdAt`만 직접 선언하게 하거나, 정말 공통화가 필요하면 `CreatedTimeEntity(createdAt only)`를 별도로 두세요. 단, 현재 문서의 "2단 단순화" 목표를 유지하려면 직접 선언이 더 단순합니다.

### `docs/superpowers/specs/2026-06-04-g2-common-exception-design.md:15` — 성공 기준이 보안 예외 범위 경계와 충돌함

성공 기준은 "모든 실패 응답이 단일 `@RestControllerAdvice`를 통해" 나간다고 되어 있지만, 같은 문서의 90-94라인은 인증/인가 실패가 Security filter 레벨에서 발생하므로 G2에서는 advice 밖이라고 정의합니다.

영향: 구현자가 성공 기준을 문자 그대로 따르면 `AuthenticationException`/`AccessDeniedException`까지 advice로 끌어오려는 잘못된 설계를 할 수 있고, 반대로 범위 경계를 따르면 성공 기준 2번을 만족하지 못한 것처럼 보입니다.

제안: 성공 기준을 "G2 범위의 컨트롤러 계층 실패는 단일 `@RestControllerAdvice`를 통해 RFC 7807 형식으로 나간다. 보안 필터 계층 실패는 #4에서 같은 `ErrorResponse`/`ErrorCode`를 재사용한다"로 좁히세요.

### `docs/superpowers/specs/2026-06-04-g2-common-exception-design.md:56` — `@DataJpaTest` 전략에 실제 datasource 구성이 빠져 있음

현재 프로젝트는 H2 의존성이 없고, `application.yml`의 기본 datasource는 PostgreSQL 컨테이너 호스트(`postgres`)와 필수 비밀번호 환경변수를 전제로 합니다. 기존 컨텍스트 테스트도 `TestcontainersConfiguration`을 `@Import`해서 DB/Redis를 주입합니다. 그런데 BaseEntity 감사 테스트 전략은 Flyway 비활성화와 `create-drop`만 언급하고, Testcontainers import나 datasource override가 없습니다.

영향: 이 설계대로 `@DataJpaTest`를 작성하면 테스트 슬라이스가 사용할 DB를 찾지 못하거나, 로컬에 `postgres` 호스트가 없어서 실패할 가능성이 큽니다. G2 성공 기준의 `./gradlew build` green을 직접 깨는 유형입니다.

제안: `BaseEntityAuditingTest` 전략에 `@Import(TestcontainersConfiguration.class)` 또는 동등한 PostgreSQL Testcontainer datasource 주입을 명시하세요. 테스트 전용 엔티티 때문에 Flyway만 끄고 `ddl-auto=create-drop`으로 격리하는 방향은 맞습니다.

## Positive

- `createdBy`/`updatedBy`를 `Long`으로 둔 결정은 `global -> domain` 역의존을 피하므로 현재 `ArchitectureTest`와 잘 맞습니다.
- `ErrorCode`를 단일 진실 공급원으로 두고 Security 계층은 후속 이슈에서 재사용하게 한 범위 분리는 타당합니다.
- `PagedModel` 채택은 현재 로컬 Spring Data Commons 4.0.5에서 `content`와 `page` JSON 속성을 제공하므로 스펙의 목록 응답 모양과 맞습니다.
