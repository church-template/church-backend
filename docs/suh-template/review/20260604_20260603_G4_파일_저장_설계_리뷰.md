# G4 파일 저장 설계 리뷰

대상: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md`

## Critical

없음.

## Major

### 1. 확장자 보존 정책이 원본 파일명 비사용 원칙을 완전히 만족하지 못함

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:32`, `:106`, `:134`
- 문제: 문서는 원본 파일명을 디스크 경로에 쓰지 않는다고 하지만, 확장자는 원본 파일명에서 추출해 경로에 붙인다. 이때 확장자 추출 규칙이 명시되어 있지 않아 `a.jpg/../evil`, `a..//x`, `a.` 같은 입력에서 슬래시, `..`, 빈 확장자, 비정상 문자가 저장 키에 섞일 수 있다.
- 영향: `startsWith(root)` 검증이 외부 탈출은 막더라도, 저장 키가 `yyyy/MM/{uuid}.{ext}` 계약을 벗어나거나 루트 내부의 예기치 않은 하위 경로를 만들 수 있다. 원본명 경로우회 테스트도 "안전"만 검증하면 이 케이스를 놓친다.
- 제안: 확장자는 먼저 basename만 대상으로 추출하고, 소문자 ASCII `[a-z0-9]+` 정도의 허용 목록을 통과한 경우에만 붙이도록 설계에 명시한다. 통과하지 못하면 확장자 없이 저장하거나 `INVALID_INPUT_VALUE` 중 하나로 고정한다. 테스트에는 슬래시/점 세그먼트가 포함된 원본명에서도 반환 키가 정확히 `yyyy/MM/<uuid>(.<safe-ext>)?` 패턴을 유지하는 검증을 추가한다.

### 2. `load`/`delete`가 일반 파일만 대상으로 한다는 조건이 빠져 있음

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:110-116`, `:136-139`
- 문제: 현재 설계는 존재/가독 여부만 말하고 `Files.isRegularFile(target)` 검증을 명시하지 않는다. `PathResource`는 디렉터리 경로도 리소스로 만들 수 있고, `Files.deleteIfExists(target)`는 루트 내부의 빈 디렉터리도 삭제할 수 있다.
- 영향: 외부 공격자가 임의 경로를 직접 전달할 수 있는 후속 API가 붙으면 파일 저장소가 의도치 않게 디렉터리를 조회/삭제 대상으로 취급할 수 있다.
- 제안: `load`는 `Files.exists && Files.isRegularFile && Files.isReadable`일 때만 성공하고, 그 외는 `RESOURCE_NOT_FOUND`로 통일한다. `delete`는 일반 파일일 때만 삭제하고, 디렉터리/특수 파일은 삭제하지 않는다고 계약에 명시한다.

### 3. `delete`의 경로우회 처리와 idempotent 계약이 충돌함

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:21`, `:91`, `:114-116`, `:137`, `:139`
- 문제: 문서는 `delete`가 없는 파일을 조용히 무시한다고 하면서, 경로우회 입력은 거부해야 한다고 말한다. 하지만 `delete` 계약의 throws에는 `FILE_STORAGE_ERROR`만 있고, 경로우회 시 `RESOURCE_NOT_FOUND`를 던질지 no-op으로 볼지 불명확하다.
- 영향: 구현자와 테스트가 서로 다른 해석을 할 수 있다. 특히 테스트 #8은 "거부"라고만 되어 있어 예외 코드까지 고정하지 못한다.
- 제안: 둘 중 하나로 명확히 고정한다. 보안 이벤트를 호출자에게 드러내지 않으려면 `delete`의 루트 외부 경로는 no-op으로 보고 로그만 남긴다. 명시적 거부를 원하면 인터페이스 Javadoc에 `RESOURCE_NOT_FOUND` 또는 `INVALID_INPUT_VALUE`를 throws로 추가하고 테스트 기대값도 고정한다.

## Minor

### 1. `FileProperties.baseUrl`은 G4에서 미사용인데 필수 검증 대상임

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:35`, `:45`, `:58`
- 현재 판단: `application.yml`에 기본값이 있어 기동에는 문제가 없지만, 스토리지 구현 자체는 `baseUrl`을 사용하지 않는다.
- 제안: 유지한다면 "G4에서는 바인딩/검증만 하며 LocalFileStorage 동작에는 관여하지 않는다"를 산출물 섹션에 한 줄 더 명시한다. 아니면 후속 media 이슈에서 별도 properties로 옮기는 선택지도 있다.

### 2. I/O 실패 후 부분 파일 정리 정책이 없음

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:107-108`
- 현재 판단: 실패를 `FILE_STORAGE_ERROR`로 매핑하는 것은 맞지만, `transferTo`/`Files.copy` 도중 실패하면 일부 파일이 남을 수 있다.
- 제안: 구현 범위를 키우지 않으려면 "실패 시 best-effort로 target 삭제" 정도만 설계에 추가한다. 더 엄격하게 하려면 임시 파일에 쓴 뒤 `Files.move(..., ATOMIC_MOVE)`를 고려할 수 있으나, 이번 범위에서는 best-effort 삭제가 충분하다.

### 3. `.env.example` 주석 정리 항목이 테스트/산출물과 분리되어 있음

- 위치: `docs/superpowers/specs/2026-06-04-g4-file-storage-design.md:71`
- 현재 판단: 실제 `.env.example`에는 `FILE_MAX_SIZE` 주석이 없어 문서의 "함께 다듬는다"는 구현 산출물에 포함될 가능성이 높다.
- 제안: 산출물 섹션의 수정 파일에 `.env.example`을 명시하거나, 비범위로 둘지 결정한다.

## Positive

- G4 범위를 `global/storage` 인프라로 제한한 결정은 현재 로드맵의 Phase 1 역할과 잘 맞는다.
- `media` 도메인, member FK, 서빙 API, MIME 정책을 후속으로 분리한 판단은 의존성 순서를 흐리지 않는다.
- 현재 프로젝트의 `application.yml`, `.env.example`, `SecurityConfig`의 `@EnableConfigurationProperties` 패턴, `ArchitectureTest`와 대체로 정합성이 있다.
- `@TempDir` 기반 단위 테스트로 Spring 컨텍스트 없이 핵심 I/O를 검증하려는 방향은 빠르고 충분히 실용적이다.

## 리뷰 요약

- 전체 평가: Request Changes
- 이슈 통계: Critical 0개 / Major 3개 / Minor 3개
- 핵심 개선 사항:
  1. 원본 파일명에서 추출하는 확장자 sanitize 규칙을 명확히 한다.
  2. `load`/`delete` 대상은 일반 파일로 제한한다.
  3. `delete`의 경로우회 입력 처리와 idempotent 계약을 하나로 고정한다.
