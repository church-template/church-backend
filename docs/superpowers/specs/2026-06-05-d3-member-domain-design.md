## 1. 목표와 범위

회원 가입 정보 관리·약관 동의/재동의·교인 승인(역할 부여)·관리자 회원관리를 구현한다. 본인 확인은 이메일 인증이 아니라 **관리자의 `MEMBER` 역할 부여**로 대체한다(서로 아는 교회 특성).

### 포함 (D3)
- `members`·`member_roles` 스키마(`V3` 마이그레이션) + 이연된 `positions` FK(`ON DELETE SET NULL`)
- `GET/PATCH /api/members/me` — DB 최신값 조회, 이름·전화번호·비밀번호·이메일 수정(전화번호 중복 체크)
- `GET/PATCH /api/members/me/agreements` — 동의 상태 조회·재동의 제출
- 관리자: 회원 목록·상세(`MEMBER_MANAGE`), `PATCH /api/admin/members/{uuid}`, `reset-password`
- 역할 부여/회수(`ROLE_MANAGE`, 위계·자기보호·마지막 SUPER_ADMIN 가드). `MEMBER` 부여 = 교인 승인
- 약관 일괄 리셋(`POST /api/admin/agreements/reset`)
- 최초 SUPER_ADMIN 부트스트랩(기동 시 env 러너, 멱등)
- `RoleService.delete()` 블로킹 삭제 완성(`member_roles` 참조 시 409)

### 제외 (D4 auth로 이연)
- `signup`·`login`·`refresh`·`logout` 엔드포인트.
- 단 `Member.create()`(uuid 생성·BCrypt 저장), `MemberRepository.findByPhoneAndDeletedAtIsNull`, phone 정규화 유틸은 **D3에서 만들어 D4가 재사용**한다.
- D3에서 일반 회원을 생성하는 경로는 부트스트랩 러너와 테스트뿐이다(가입 API가 D4라서). D2(role)에 회원이 없던 것과 같은 단계적 상황.

## 2. 데이터 모델 — `V3__create_members.sql`

```sql
CREATE TABLE members (
  id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uuid          UUID         NOT NULL,
  phone         VARCHAR(20)  NOT NULL,     -- 숫자만 정규화 저장(예: 01012345678)
  name          VARCHAR(50)  NOT NULL,
  password      VARCHAR(255) NOT NULL,     -- BCrypt 해시
  email         VARCHAR(255),              -- 선택, 유니크 없음
  position_id   BIGINT,
  terms_agreed   BOOLEAN NOT NULL DEFAULT FALSE,
  privacy_agreed BOOLEAN NOT NULL DEFAULT FALSE,
  agreed_at      TIMESTAMP,
  created_at    TIMESTAMP NOT NULL,
  updated_at    TIMESTAMP,
  deleted_at    TIMESTAMP,
  CONSTRAINT fk_members_position FOREIGN KEY (position_id)
      REFERENCES positions(id) ON DELETE SET NULL   -- V1에서 이연된 FK(직분 물리 삭제 대응)
);
CREATE UNIQUE INDEX uq_members_uuid         ON members (uuid);
CREATE UNIQUE INDEX uq_members_phone_active ON members (phone) WHERE deleted_at IS NULL;  -- 번호 재활용

CREATE TABLE member_roles (
  member_id BIGINT NOT NULL,
  role_id   BIGINT NOT NULL,
  PRIMARY KEY (member_id, role_id),
  CONSTRAINT fk_mr_member FOREIGN KEY (member_id) REFERENCES members(id) ON DELETE CASCADE,
  CONSTRAINT fk_mr_role   FOREIGN KEY (role_id)   REFERENCES roles(id)   ON DELETE RESTRICT  -- V2 NOTE 계약(앱 레벨 블로킹)
);
CREATE INDEX idx_member_roles_member ON member_roles (member_id);
CREATE INDEX idx_member_roles_role   ON member_roles (role_id);
```

- 인덱스는 스펙 §6 그대로: `(phone) WHERE deleted_at IS NULL` UNIQUE(로그인 조회 + 번호 재활용), `(uuid)` UNIQUE(외부 식별자), `member_roles (member_id)`·`(role_id)`.
- **빌드-그린 불변식**: 마이그레이션(테이블)을 엔티티보다 **먼저** 커밋한다. SB4 `ddl-auto: validate`는 테이블이 없는 채 엔티티가 생기면 "missing table"로 전 `@SpringBootTest`를 깬다(직분·RBAC에서 확인된 함정).

## 3. 엔티티

### Member
기존 두 베이스 클래스 중 어디에도 정확히 맞지 않는다:
- `BaseTimeEntity`(created_at만) — 회원은 `deleted_at`(소프트삭제)·`updated_at`도 필요.
- `BaseEntity`(full audit + version) — 회원은 `@Version`(스펙 낙관락 대상에 회원 **제외**), `created_by`/`updated_by`(자가 가입이라 감사자 부재)가 불필요.

→ **`BaseTimeEntity` 상속 + `updatedAt`(`@LastModifiedDate`)·`deletedAt` 직접 선언**. 식별자 불변(스펙 §3): `id`(BIGINT 내부 FK)·`uuid`(외부 식별자, JWT sub) 불변, `phone`/`name`/`email`/`password`는 가변.

```java
@Entity @Table(name = "members")
public class Member extends BaseTimeEntity {
    @Id @GeneratedValue(strategy = IDENTITY) Long id;
    @Column(nullable = false, columnDefinition = "uuid") UUID uuid;   // 경계에서 String 변환(토큰 sub·경로 변수)
    @Column(nullable = false) String phone;                          // 숫자만 정규화 저장
    @Column(nullable = false) String name;
    @Column(nullable = false) String password;                       // BCrypt
    String email;                                                    // nullable
    @ManyToOne(fetch = LAZY) @JoinColumn(name = "position_id") Position position;  // nullable
    boolean termsAgreed; boolean privacyAgreed; LocalDateTime agreedAt;
    @LastModifiedDate LocalDateTime updatedAt;
    LocalDateTime deletedAt;
    @ManyToMany(fetch = LAZY) @JoinTable(name = "member_roles", ...) Set<Role> roles;
}
```

도메인 메서드(불변 패턴, 세터 금지):
- `static Member create(phone, name, encodedPassword, email, position, termsAgreed, privacyAgreed)` — `uuid = UUID.randomUUID()`. **생성 invariant: 필수 약관 2종이 모두 true가 아니면 `INVALID_INPUT_VALUE`로 거부**하고, 성립 시 `agreedAt = now`. 회원은 항상 "동의된 상태"로만 생성된다(미동의는 사후 `resetAgreement`로만 발생). D4 signup이 이 팩토리를 재사용하며(요청 boolean을 그대로 전달, DTO 검증과 이중 방어), 부트스트랩은 운영자 동의로 `(true, true)`를 넘긴다.
- `updateProfile(name, phone, email)` — null 미변경(부분 수정).
- `changePassword(encoded)` / `agree()`(필수 2종 true + agreedAt=now) / `resetAgreement(target)` / `grantRole(role)` / `revokeRole(role)` / `softDelete()`.

### 권한 플래트닝
`member.getRoles()` → 각 `role.getPermissions()` → distinct 권한명 union; `maxPriority = max(role.priority)` 또는 0. `@EntityGraph(attributePaths = {"roles", "roles.permissions"})`로 `/me`·관리자 조회 N+1 회피. (이 플래트닝 로직은 D4 토큰 발급도 재사용.)

### MemberRepository 주요 쿼리
- `findByUuidAndDeletedAtIsNull(UUID)` / `findByPhoneAndDeletedAtIsNull(String)`(D4 로그인 재사용)
- `existsByPhoneAndDeletedAtIsNull(String)` — 가입 시 중복 체크
- `existsByPhoneAndDeletedAtIsNullAndIdNot(String, Long)` — **번호 변경 시 자기 자신 제외 중복 체크**(PATCH /me·관리자 PATCH). 정규화 후 기존 phone과 동일하면 체크 자체를 생략(불필요한 409 오판 방지). 선검사를 빠져나간 동시 변경은 partial unique 충돌을 `saveAndFlush` try/catch(`DataIntegrityViolationException`)로 잡아 `DUPLICATE_RESOURCE`(409) 백스톱.
- `findAllByDeletedAtIsNull(Pageable)` — 관리자 목록
- `countByRoles_NameAndDeletedAtIsNull("SUPER_ADMIN")` — 마지막 SUPER_ADMIN 가드
- `existsByRoles_NameAndDeletedAtIsNull("SUPER_ADMIN")` — **부트스트랩 멱등 판정(활성 회원 기준)**. soft-delete된 SUPER_ADMIN만 남은 경우엔 활성 최고관리자가 없는 것이므로 새 계정을 생성해야 한다 → 마지막-SUPER_ADMIN 가드와 동일하게 `deleted_at IS NULL` 기준으로 통일.
- `@Modifying UPDATE Member m SET m.termsAgreed=false WHERE m.deletedAt IS NULL`(target별 2쿼리 또는 분기) — 약관 일괄 리셋

## 5. API 명세

### 5.1 본인 (인증)
- `GET /api/members/me` → `MeResponse { uuid, name, phone, email, position(한글 name|null), roles:[{name,description}], permissions:[...], maxPriority, termsAgreed, privacyAgreed, agreedAt }`. **DB 최신값**(토큰 lag 회피).
- `PATCH /api/members/me` ← `{ name?, phone?, password?, email? }`. phone 변경 시 정규화 후 **자기 자신 제외** 활성 회원과 중복이면 `409 DUPLICATE_RESOURCE`(동일 번호 유지 시 체크 생략 — §4). password 변경 시 **최소 길이(≥8) 검증 후 BCrypt 인코딩**(복잡도 강제 없음 — `multi-church-template` 룰). 부분 수정.
  - **가정**: 현재 비번 미요구(인증 세션 + 고령 친화·복잡 정책 배제 원칙).
- `GET /api/members/me/agreements` → `{ termsAgreed, privacyAgreed, agreedAt }`.
- `PATCH /api/members/me/agreements` ← `{ termsAgreed, privacyAgreed }`. 필수 2종 모두 true여야 성립(아니면 `400 INVALID_INPUT_VALUE`, signup과 동일 규칙), `agreed_at=now`.

### 5.2 관리자 회원관리
- `GET /api/members?page&size&sort`(`MEMBER_MANAGE`) → 공통 페이지 봉투. 카드 = `MemberCardResponse { uuid, name, phone, position, roles:[name], approved(=MEMBER 보유), createdAt }`.
- `GET /api/members/{uuid}`(`MEMBER_MANAGE`) → `MemberDetailResponse`(카드 + email·permissions·약관 상태).
- `PATCH /api/admin/members/{uuid}`(`MEMBER_MANAGE`) ← `{ name?, phone?, email? }`. 전화번호 갱신 구제(중복 체크 동일). 비번은 reset-password로만.
- `POST /api/admin/members/{uuid}/reset-password`(`MEMBER_MANAGE`, 바디 없음) → 서버가 **`SecureRandom`으로 최소 길이(≥8) 임시 비번 생성** → BCrypt 저장 → `ResetPasswordResponse { temporaryPassword }`로 **평문 1회만 반환**(관리자가 교인에게 대면 전달). 임시 비번은 **로그·예외 메시지에 절대 기록하지 않는다**.

### 5.3 역할 부여/회수 (`ROLE_MANAGE`, 위계 검증)
- `POST /api/admin/members/{uuid}/roles` ← `{ roleId }` → 부여. `MEMBER` 부여 = 교인 승인.
- `DELETE /api/admin/members/{uuid}/roles/{roleId}` → 회수.
- 가드 적용 순서(모두 `RoleHierarchyValidator` 재사용, 위반 시 `403 ACCESS_DENIED`):
  1. 대상 회원(uuid)·역할(roleId) **리소스 조회** — 자체가 없으면 `404 RESOURCE_NOT_FOUND`.
  2. `validateNotSelf(requester.id, target.id)` — **항상** 적용(요청자 `mid` 클레임 vs uuid 조회 대상). 정체성 기반 가드라 보유 여부·no-op과 무관하게 자기 역할 부여/회수를 금지.
  3. `validateAssignable(requester.maxPriority, role.priority)` — **항상** 적용. 대상 역할 priority가 요청자 maxPriority 초과면 거부(보유 여부·no-op과 무관한 "그 역할을 조작할 권한" 가드, 같은 레벨 허용).
  4. **회수이면서 대상이 실제 SUPER_ADMIN을 보유 중일 때만** `validateNotLastSuperAdmin(true, activeSuperAdminCount)` — 마지막 활성 SUPER_ADMIN 회수 금지. (세 가드 중 이것만 "실제 회수 결과"에 의존.)
- 멱등 시맨틱: 위 2·3 가드를 통과한 뒤 — 이미 보유한 역할 재부여는 무해 성공(200), 보유하지 않은 역할 회수는 무해 성공(204, 멱등 DELETE).
- 효과는 **다음 토큰 갱신 시 반영**(스펙대로). Redis 강제 만료는 범위 외.

### 5.4 약관 일괄 리셋 (`MEMBER_MANAGE`)
- `POST /api/admin/agreements/reset` ← `{ "target": "terms" | "privacy" }` → 미삭제 회원 전체 해당 플래그 false 일괄 UPDATE. 잘못된 target은 `400`.
- **`agreed_at` 정책**: 단일 timestamp는 "마지막 동의 행위 시각"을 뜻한다. 일괄 리셋은 플래그만 false로 바꾸고 **`agreed_at`은 변경하지 않는다**(NULL로 만들지 않음). 재동의가 트리거되는 기준은 "필수 플래그 중 false 존재"이지 timestamp가 아니므로, `agreed_at`은 마지막 동의 시각의 정보값으로 유지된다. 재동의 제출(`PATCH /me/agreements`) 시에만 `agreed_at = now`로 갱신.

## 6. SUPER_ADMIN 부트스트랩 — `SuperAdminInitializer implements ApplicationRunner`

- 설정: `admin.bootstrap.{phone,name,password}` ← `application.yml`에서 `${ADMIN_PHONE:}`·`${ADMIN_NAME:}`·`${ADMIN_PASSWORD:}`(`@ConfigurationProperties`).
- 동작: 기동 시 **`existsByRoles_Name("SUPER_ADMIN")`가 false(=SUPER_ADMIN 회원 0명)이고 env가 채워졌을 때만** BCrypt 해시로 회원 1명 생성 + SUPER_ADMIN 역할 링크. **멱등**(이미 있으면 skip). env 미설정이면 skip + warn 로그.
- 위치: `members` + `roles`를 함께 참조하므로 `global`이 아니라 **`domain/member`**(ArchUnit `global→domain` 금지 회피; domain→domain은 허용).
- 템플릿 철학: BCrypt 해시를 repo에 굽지 않고 교회별 `.env`로 분리. `.env.example`에 `ADMIN_PHONE`·`ADMIN_NAME`·`ADMIN_PASSWORD` 주석 추가.

## 7. 기존 코드 수정 (최소 침습)

1. **`RoleService.delete()` 블로킹 삭제 완성**: `roleRepository.delete(role); roleRepository.flush();`를 try/catch(`DataIntegrityViolationException`)로 감싸 → `BusinessException(ErrorCode.ROLE_IN_USE)`. `member_roles.role_id` FK RESTRICT(DB 백스톱)를 친절한 409로 번역. role→member **코드 결합 회피**(기존 `saveAndFlush` 백스톱 패턴 재사용). `RoleService`/`RoleRepository`의 `NOTE(#8)` 주석 제거.
2. **`ErrorCode.ROLE_IN_USE`(409) 추가**: 회원에 할당된 역할 삭제 차단 전용. 기존 코드로 매핑 불가한 신규 상황이라 신설(`MEDIA_IN_USE`는 미디어 전용, `DUPLICATE_RESOURCE`는 의미 불일치).
3. **`PasswordEncoder`(`BCryptPasswordEncoder`) 빈 추가**: `global/config`(예: `SecurityConfig` 또는 별도 `PasswordConfig`). reset-password·비번 변경·부트스트랩이 사용. D4 login도 재사용.

## 8. 테스트 전략 (커버리지 80%+)

기존 하네스 재사용: `@SpringBootTest` + `@AutoConfigureMockMvc` + `@Import(TestcontainersConfiguration)`, `JwtTokenProvider.issueAccess()`로 요청자 토큰 직접 발급(로그인 API는 D4라 미존재). `@AfterEach`로 테스트 생성 회원만 정리(시드 역할 보존).
