# G6(#7) 역할·권한(RBAC) 도메인 + 시드 — 설계

> 상태: 설계 확정 (브레인스토밍 합의)
> 선행: G3(#4, 보안 기반·`RoleHierarchyValidator`)·G5(#6, 첫 Flyway 마이그레이션 패턴)
> 후행: #8 회원(members 테이블·member_roles·SUPER_ADMIN 계정·역할 부여/회수)

## 1. 배경 & 범위

스펙 §3·§4·§5.4의 Discord식 동적 RBAC 중 **역할(Role)·권한(Permission) 코어**를 구현한다. 인가는 권한(Permission) 단위로 판단하고, 역할은 권한 묶음 + priority(위계)를 갖는 그릇이다.

### 1.1 이번 이슈(#7)에 포함

- 마이그레이션 `V2__create_rbac.sql`: `permissions`, `roles`, `role_permissions` 테이블 + 시드.
- 시드: permission 12종, role 4종(SUPER_ADMIN·ADMIN·MEMBER·USER), role_permissions 매트릭스.
- 역할 API(§5.4): 목록 / 생성 / 수정 / 삭제 / 권한 일괄 설정 / 전체 권한 목록.
- 역할 수정·삭제·권한변경 시 위계 검증(`RoleHierarchyValidator` 재사용).
- **글로벌 위계 규칙을 `<=`(같은 레벨 허용)으로 변경** — `RoleHierarchyValidator`·테스트·스펙 §4.3 동시 수정(아래 §6.3, §9).

### 1.2 #8(회원)로 이연 — members 테이블 의존

`members`가 아직 없으므로(직분 #6이 `position_id` FK를 #8로 미룬 선례와 동일) 다음은 #8에서 구현한다.

- `member_roles` 조인 테이블·`members` FK. **삭제 계약:** `member_roles.role_id → roles(id)`는 RESTRICT(cascade 아님) + 앱 레벨 **블로킹 삭제** — 회원이 1명이라도 보유한 역할은 `DELETE /roles/{id}`를 거부(409). 미디어 블로킹 삭제(409 `MEDIA_IN_USE`) 철학과 일치, 회원이 권한을 조용히 잃는 것 방지. (대안 `ON DELETE CASCADE`는 무음 권한 박탈 위험으로 미채택.) #8에서 할당 검사 + 테스트 추가.
- **최초 SUPER_ADMIN 계정 1개 시드**(전화번호·임시 비밀번호).
- 회원-역할 부여/회수 엔드포인트(`POST`/`DELETE /api/admin/members/{uuid}/roles...`, §5.2).
- `validateAssignable`(회원에게 역할 부여 시)·`validateNotSelf`·`validateNotLastSuperAdmin` **사용처**. (메서드 자체는 G3에 이미 존재; #7에선 `validateMutable`만 호출)

## 2. 핵심 설계 결정 (Decision Log)

| # | 결정 | 근거 |
|---|---|---|
| D1 | 범위 = 역할·권한 코어만, member 결합부는 #8로 분리 | members 미존재. FK 깨짐 방지. #6이 members FK를 #8로 미룬 선례와 일치. §5.4엔 회원-역할 부여가 없음(그건 §5.2=#8) |
| D2 | Permission 표현 = **DB-only**(JPA 엔티티 + Flyway 시드). enum 미도입 | §2.1 단순성. `@PreAuthorize` 문자열은 어차피 enum으로 못 막음. 시드 정합성은 **시드 무결성 테스트**로 확보. Position(순수 JPA+Flyway) 패턴과 일치 |
| D3 | 위계 규칙 = **`<=`(같은 레벨 허용)**, 전면 적용 | 사용자 결정. "자기 레벨과 같은 수준까지 부여/관리, 강등 가능." 스펙 §4.3·`RoleHierarchyValidator` 변경 동반(§9) |
| D4 | ADMIN 시드 = **전 12종(ROLE_MANAGE 포함)** | 사용자 결정. ADMIN이 회원에게 역할 부여(ROLE_MANAGE, §5.2 325·326행) 가능해야 함 |
| D5 | `PUT /roles/{id}/permissions` 바디 = **권한 name 리스트** `["SERMON_WRITE", ...]` | 영문 키라 DB id 비의존·포터블·자가문서화. 미지 키 → `INVALID_INPUT_VALUE` |
| D6 | 역할 목록 = **비페이징 평배열, priority DESC** | 소규모 어드민 마스터 목록. 직분(#6) 마스터 목록 평배열 선례 |
| D7 | `roles`=물리 삭제(`BaseTimeEntity`), `permissions`=plain `@Entity` | 스펙 §3.2 테이블에 soft-delete/version/타임스탬프 없음. Position과 동일 |
| D8 | role↔permission = `@ManyToMany Set<Permission>` (별도 조인 엔티티 X) | 조인 테이블에 추가 컬럼 없음. PUT=전체 교체 시맨틱에 자연스러움 |
| D9 | 신규 ErrorCode 없음 | 기존 `DUPLICATE_RESOURCE`/`RESOURCE_NOT_FOUND`/`ACCESS_DENIED`/`INVALID_INPUT_VALUE` 재사용 |
| D10 | role name = trim 정규화 + unique, **영어 `@Pattern` 미적용** | role name은 인가 키 아님(라벨). 스펙 §3.2가 한글 예시(`콘텐츠관리자`) 허용 → 영어강제는 §3.2와 충돌. 코드참조 seed 역할만 영어 |
| D11 | 역할 삭제 ↔ `member_roles` = **블로킹 삭제**(RESTRICT + 앱 409) | 미디어 블로킹 삭제 철학 일치, 무음 권한 박탈 방지. #8 계약(§1.2). cascade 미채택 |

> **FYI(스펙 한계, #8에서 재검토):** 스펙의 `ROLE_MANAGE`는 "역할 정의 CRUD"(§5.4)와 "회원에게 역할 부여"(§5.2)를 **한 권한으로** 묶는다. 따라서 향후 "성도 관리자" 역할에 ROLE_MANAGE를 부여하면 교인 승인뿐 아니라 역할 정의 변경까지 가능해진다. "부여만 허용/정의는 불가"로 분리하려면 ROLE_MANAGE를 둘로 쪼개야 하며(13번째 permission → §3.3 변경), 이는 회원-역할 부여를 실제 구현하는 **#8에서 필요 시** 재검토한다. #7 시드는 스펙대로 12종을 유지한다.

## 3. 데이터 모델

### 3.1 마이그레이션 `V2__create_rbac.sql`

```sql
-- 권한(고정 카탈로그). 타임스탬프 없음(스펙 §3.2).
CREATE TABLE permissions (
    id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name        VARCHAR(50)  NOT NULL,
    description VARCHAR(255),
    CONSTRAINT uq_permissions_name UNIQUE (name)
);

-- 역할(관리자가 동적 생성). created_at만, soft-delete/version 없음 → 물리 삭제.
CREATE TABLE roles (
    id          BIGINT  GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name        VARCHAR(50)  NOT NULL,
    priority    INTEGER      NOT NULL,
    is_system   BOOLEAN      NOT NULL DEFAULT FALSE,
    description VARCHAR(255),
    created_at  TIMESTAMP    NOT NULL,
    CONSTRAINT uq_roles_name UNIQUE (name)
);
CREATE INDEX idx_roles_priority ON roles (priority);

-- 역할 ↔ 권한 (다대다). 역할 물리 삭제 시 링크 동반 삭제.
CREATE TABLE role_permissions (
    role_id       BIGINT NOT NULL,
    permission_id BIGINT NOT NULL,
    PRIMARY KEY (role_id, permission_id),
    CONSTRAINT fk_rp_role       FOREIGN KEY (role_id)       REFERENCES roles (id)       ON DELETE CASCADE,
    CONSTRAINT fk_rp_permission FOREIGN KEY (permission_id) REFERENCES permissions (id) ON DELETE RESTRICT
);

-- (시드는 §4 참고)
-- NOTE(#8 members): member_roles(member_id, role_id) 조인 테이블과 members FK,
--   최초 SUPER_ADMIN 계정 시드는 members 마이그레이션에서 추가한다.
--   member_roles.role_id -> roles(id)는 RESTRICT(앱 레벨 블로킹 삭제, §1.2). 할당된 역할 삭제는 #8에서 409로 가드.
```

### 3.2 엔티티 (`domain/role`)

**`Permission`** — plain `@Entity`(베이스 미상속).
- `Long id`, `String name`(unique, len 50), `String description`(len 255).
- `static Permission of(String name, String description)` (테스트·명료성용).

**`Role extends BaseTimeEntity`** (created_at만).
- `Long id`, `String name`(unique), `int priority`, `boolean isSystem`, `String description`.
- `@ManyToMany(fetch = LAZY) @OrderBy("name ASC") @JoinTable(name="role_permissions", ...) Set<Permission> permissions = new LinkedHashSet<>();` — `@OrderBy`로 응답 순서를 name ASC로 고정(Set 반환 순서 불안정 → API 테스트·프론트 표시 흔들림 방지).
- `static Role create(String name, int priority, String description)` — **`isSystem`는 항상 false**(시스템 역할은 시드로만 생성). 
- `void update(String name, Integer priority, String description)` — 각 인자 null이면 미변경(부분 수정, Position 패턴).
- `void replacePermissions(Collection<Permission> next)` — `permissions.clear(); permissions.addAll(next);` (관리 엔티티 in-place, Position.update와 동일 방식).

## 4. 시드 데이터 (V2 INSERT)

### 4.1 permissions (12종, 영문 name + 한글 description, §3.3)

`SERMON_WRITE`(설교 등록/수정/삭제), `NOTICE_WRITE`(공지 등록/수정/삭제), `EVENT_WRITE`(일정 등록/수정/삭제), `DEPT_WRITE`(교구/부서 등록/수정/삭제), `MEMBER_MANAGE`(회원 조회/관리), `ROLE_MANAGE`(역할·권한 관리), `POSITION_MANAGE`(직분 관리), `MEDIA_MANAGE`(미디어 업로드/조회/삭제), `TAG_MANAGE`(태그 추가/수정/삭제), `GALLERY_WRITE`(갤러리 업로드/수정/삭제), `GALLERY_VIEW`(갤러리 조회 — 회원 전용 열람), `BULLETIN_WRITE`(주보 업로드/수정/삭제).

### 4.2 roles (4종) + role_permissions 매트릭스

| name | priority | is_system | description | 보유 권한 |
|---|---|---|---|---|
| SUPER_ADMIN | 1000 | true | 최고 관리자 | **전 12종** |
| ADMIN | 900 | true | 관리자 | **전 12종 (ROLE_MANAGE 포함)** |
| MEMBER | 100 | false | 교인 | `GALLERY_VIEW` |
| USER | 0 | true | 일반 사용자 | (없음) |

### 4.3 시드 SQL (id 비의존 — name 조인)

```sql
INSERT INTO permissions (name, description) VALUES ('SERMON_WRITE', '설교 등록/수정/삭제'), ... ; -- 12행

INSERT INTO roles (name, priority, is_system, description, created_at) VALUES
  ('SUPER_ADMIN', 1000, TRUE,  '최고 관리자',   now()),
  ('ADMIN',        900, TRUE,  '관리자',        now()),
  ('MEMBER',       100, FALSE, '교인',          now()),
  ('USER',           0, TRUE,  '일반 사용자',   now());

-- SUPER_ADMIN·ADMIN = 전 권한
INSERT INTO role_permissions (role_id, permission_id)
SELECT r.id, p.id FROM roles r CROSS JOIN permissions p
WHERE r.name IN ('SUPER_ADMIN', 'ADMIN');

-- MEMBER = GALLERY_VIEW
INSERT INTO role_permissions (role_id, permission_id)
SELECT r.id, p.id FROM roles r JOIN permissions p ON p.name = 'GALLERY_VIEW'
WHERE r.name = 'MEMBER';
-- USER = 없음
```

## 5. API 명세 (§5.4)

모든 엔드포인트 `@PreAuthorize("hasAuthority('ROLE_MANAGE')")`. 경로는 `/api/admin/**`(SecurityConfig가 이미 authenticated 처리 — **SecurityConfig 무수정**).

| 메서드 | 경로 | 상태 | 설명 |
|---|---|---|---|
| GET | `/api/admin/roles` | 200 | 역할 목록. **비페이징 평배열, priority DESC**. 각 항목에 permissions 포함 |
| POST | `/api/admin/roles` | 201 | 역할 생성. body `{name, priority, description}`. `is_system=false` 강제 |
| PATCH | `/api/admin/roles/{id}` | 200 | name·priority·description 부분 수정 |
| DELETE | `/api/admin/roles/{id}` | 204 | 물리 삭제(+role_permissions cascade) |
| PUT | `/api/admin/roles/{id}/permissions` | 200 | 권한 전체 교체. body `{ "permissions": ["SERMON_WRITE", ...] }` |
| GET | `/api/admin/permissions` | 200 | 부여 가능 전체 권한 평배열(**name ASC**) |

### 5.1 요청/응답 DTO (수동 `from()` 매핑, Position 패턴)

- `RoleResponse(Long id, String name, int priority, boolean isSystem, String description, List<PermissionResponse> permissions)` — `from(Role)`. `permissions`는 **name ASC 정렬**(엔티티 `@OrderBy`로 보장).
- `PermissionResponse(Long id, String name, String description)` — `from(Permission)`.
- `RoleCreateRequest(@NotBlank @Size(max=50) String name, @NotNull Integer priority, @Size(max=255) String description)`. name은 서비스에서 **trim 정규화**(§6.2). **`@Pattern` 영어강제는 두지 않는다** — 스펙 §3.2가 role name 예시로 `콘텐츠관리자`(한글)를 들고, role name은 인가 키가 아니라 라벨이기 때문(인가는 permission 단위, JWT에도 role 미포함). 영어 코드키 규칙은 시드/코드참조 역할(SUPER_ADMIN·MEMBER 등)에만 해당.
- `RoleUpdateRequest(@Size(max=50) String name, Integer priority, @Size(max=255) String description)` — 전부 선택(부분 수정). name 비-null이면 동일하게 trim 정규화.
- `RolePermissionsRequest(@NotNull List<@NotBlank @Size(max=50) String> permissions)` — 빈 리스트 허용(전체 해제). 서비스에서 **`LinkedHashSet`로 dedup 후** 검증(중복 권한명이 "미지 권한 400"으로 오인되지 않게).

## 6. 인가 & 위계 검증

### 6.1 요청자 maxPriority 획득

컨트롤러에서 `@AuthenticationPrincipal MemberPrincipal principal`로 받아 `principal.maxPriority()`를 서비스 메서드에 인자로 전달.

### 6.2 검증 지점 (`RoleHierarchyValidator` 재사용)

- **POST(생성):** `is_system=false` 고정 + `validateAssignable(maxPriority, request.priority)` — 자기 레벨 **초과** priority 생성 거부(`<=` 허용).
- **PATCH(수정):** `validateMutable(maxPriority, role.priority, role.isSystem)`(is_system 거부 + 현재 priority 가드). `request.priority != null`이면 새 priority도 `validateAssignable(maxPriority, request.priority)`.
- **DELETE(삭제):** 역할 로드 → `validateMutable(maxPriority, role.priority, role.isSystem)` → **물리 삭제**. #7 시점엔 `member_roles` 부재라 무조건 삭제. **#8에서 "할당된 역할 삭제 금지" 가드 추가**(§1.2 계약).
- **PUT(권한 설정):** `validateMutable(maxPriority, role.priority, role.isSystem)`.
- **이름 정규화·중복:** 서비스 `normalizeName`(trim, blank→`INVALID_INPUT_VALUE`; Position 패턴) 후 `existsByName` 선검사 + `saveAndFlush` try/catch(`DataIntegrityViolationException`→`DUPLICATE_RESOURCE`) 백스톱.
- **미지/중복 권한 키:** 요청 권한명을 `LinkedHashSet`로 dedup → `permissionRepository.findByNameIn(set)` 결과 수 != dedup 집합 크기면 `INVALID_INPUT_VALUE`(중복은 무해하게 흡수, 미지 키만 거부).

### 6.3 위계 규칙 변경: `<=`(같은 레벨 허용)

G3의 `RoleHierarchyValidator`를 strict-less-than(`>=`→거부)에서 **`<=` 허용**으로 바꾼다.

```java
// 변경 전: targetPriority >= requesterMaxPriority  → 거부
// 변경 후: targetPriority >  requesterMaxPriority  → 거부 (같은 레벨 허용)
public void validateAssignable(int requesterMaxPriority, int targetPriority) {
    if (targetPriority > requesterMaxPriority) {
        throw new BusinessException(ErrorCode.ACCESS_DENIED, "대상 역할의 priority가 요청자 권한을 초과합니다");
    }
}
```

`validateMutable`은 `is_system` 가드 후 `validateAssignable` 위임 구조 그대로(규칙만 `<=`로). `validateNotSelf`·`validateNotLastSuperAdmin`은 변경 없음(#8 사용). 스펙 §4.3 문구도 "같거나 높으면 거부"→"높으면 거부"로 동기화.

## 7. 에러 처리 (신규 코드 없음, RFC 7807 전역 핸들러 경유)

| 상황 | ErrorCode | status |
|---|---|---|
| 역할명 중복 | `DUPLICATE_RESOURCE` | 409 |
| 역할/권한 미존재 | `RESOURCE_NOT_FOUND` | 404 |
| is_system·위계 위반 | `ACCESS_DENIED` | 403 |
| 검증 실패·미지 권한 키·빈 이름 | `INVALID_INPUT_VALUE` | 400 |
| ROLE_MANAGE 미보유(메서드 보안) | `ACCESS_DENIED`(인증부족) / `INVALID_TOKEN`(익명) | 403 / 401 |
