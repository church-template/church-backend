# 일정/행사(Event) 도메인 설계 — 이슈 #14 (D9)

스펙 §5.6 구현. 교회 일정·행사를 등록·조회·관리하는 단일 교회용 도메인을 추가한다.
달력·기간 조회(`?year=&month=` / `?startDate=&endDate=`)를 지원하며, 반복 일정은 1차 범위에서 제외하고 단건만 다룬다.
본문(`description`)은 마크다운 원본을 저장하고 렌더링은 프론트 책임이다(스펙 §5 본문 저장 정책).

## 0. 배경 — Notice(D8)/Sermon(D7) 패턴의 답습

Event는 Sermon·Notice가 세운 콘텐츠 도메인 패턴을 따르되, **달력 범위 조회**라는 고유점이 하나 있는 형제 도메인이다.
새로운 횡단 아키텍처 결정은 없고, 기존 자산을 그대로 재사용한다.

기존 자산 재사용(신규 횡단 컴포넌트 0개):
- `MediaReferenceProvider`(media 도메인 SPI) — 본문 `media:{id}` 참조 추적. sermon·notice에 이은 세 번째 구현체.
- `ContentTagService`(tag 도메인) — `ContentResourceType.EVENT`는 **이미 enum에 존재**. 그대로 연결.
- `BaseEntity`(global/common) — 감사/소프트삭제/낙관락 컬럼 상속.

> **`AuthorDisplayService`는 재사용하지 않는다** — Event는 작성자를 노출하지 않기 때문이다(§1 결정 참조).

## 1. 확정된 설계 결정 (브레인스토밍 합의)

| 항목 | 결정 | 근거 |
|---|---|---|
| **날짜 범위 판정** | **겹침(overlap)** — `start_at < toExclusive AND COALESCE(end_at, start_at) >= from` | 여러 날 행사(수련회 등)가 걸친 모든 달의 달력에 노출되어야 정확. `COALESCE(end_at, start_at)`로 end_at이 null인 점 이벤트가 범위 밖에서 오탐되지 않게 한다(예: 5/1 점 이벤트가 6월 조회에 끼지 않음). |
| **작성자 표시** | **노출 안 함** — 응답 DTO에 author 필드 없음 | §5.6 일정 필드가 `created_by/updated_by`를 생략하고, §5 작성자표시 정책도 대상에서 이벤트를 제외(설교·공지·갤러리·주보만 열거). 엔티티는 BaseEntity를 상속해 `version`·감사 컬럼은 보유(감사·낙관락)하되 API로는 내지 않는다. |
| **view_count** | **없음** | §5.6 일정 필드에 없고, Redis §9 조회수 카운팅 대상은 설교·공지뿐. → `get(id)`는 조회수 증가 쿼리 없음(notice/sermon과 다른 점). |
| **`?q=` 텍스트 검색** | **없음** | §5.6은 range + tag 필터만 명시. notice의 `q`에 해당하는 기능을 일부러 두지 않아 더 단순. |
| **낙관락** | `version`(@Version) + 서비스 명시적 version 비교, update/patch는 mutation 후 **`repository.flush()`** | sermon·notice에서 정착시킨 정합. flush로 응답 `version`을 post-increment(N→N+1)로 보장해 클라이언트의 다음 수정 재전송이 stale 409를 맞지 않게 한다. |
| **미디어 참조 매칭** | **경계 안전**(PG 정규식 `media:{id}($\|[^0-9])`) | 순진한 `LIKE '%media:42%'`는 `media:420`도 매칭 → 잘못된 삭제 차단. sermon·notice 선례 그대로. 매칭 대상 컬럼은 `description`. |
| **end_at / all_day** | end_at nullable(점 이벤트 허용), all_day boolean(기본 false, 표시 플래그) | start_at/end_at은 TIMESTAMP. all_day=true여도 타임스탬프는 그대로 저장하고 "종일" 표시는 프론트가 처리한다. |
| **정렬** | 기본 `start_at ASC`(`@PageableDefault`), 클라이언트 `?sort=` 덮어쓰기 허용 | 스펙 §5.6 "기본 정렬: start_at(달력·시작일 기준)". 강제 prefix 요구 문구 없음(notice 선례와 동일). |

## 2. 데이터 모델

### 2.1 마이그레이션 — `V9__create_events.sql`

> V8은 `V8__create_notices.sql`이 점유 중 → event는 **V9**.

```sql
-- 일정/행사 콘텐츠(스펙 §5.6). BaseEntity 상속, 감사/소프트삭제/낙관락 컬럼은 V7(sermons)/V8(notices) 관례를 따른다.
-- 본문 description은 마크다운 원본(TEXT), 본문 내 이미지는 media:{id}로 참조(스펙 §5 본문 정책).
CREATE TABLE events (
    id          BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title       VARCHAR(200) NOT NULL,
    description TEXT,                                  -- 마크다운 원본, media:{id} 참조
    location    VARCHAR(200),                          -- 장소(선택)
    start_at    TIMESTAMP    NOT NULL,                 -- 시작 일시
    end_at      TIMESTAMP,                             -- 종료 일시(선택, 점 이벤트면 null)
    all_day     BOOLEAN      NOT NULL DEFAULT FALSE,   -- 종일 표시 플래그
    created_at  TIMESTAMP    NOT NULL,                 -- BaseEntity @CreatedDate
    updated_at  TIMESTAMP,                             -- BaseEntity @LastModifiedDate
    created_by  BIGINT       REFERENCES members (id),  -- 최초 작성자(audit, 미노출)
    updated_by  BIGINT       REFERENCES members (id),  -- 마지막 수정자(audit, 미노출)
    deleted_at  TIMESTAMP,                             -- soft delete
    version     BIGINT       NOT NULL DEFAULT 0        -- 낙관적 락
);

-- 기본 정렬·범위 조회 = start_at, 미삭제만(스펙 §6 부분 인덱스).
CREATE INDEX idx_events_start_at ON events (start_at) WHERE deleted_at IS NULL;
```

감사 컬럼 타입은 V3(members)·V7(sermons)·V8(notices)의 관례(`TIMESTAMP`, `BIGINT`)를 따른다. `ddl-auto: validate`이므로
엔티티 매핑과 정확히 일치해야 한다(빌드로 검증). `created_by/updated_by`는 미노출이지만 BaseEntity 일관성·감사 목적으로 유지한다.

### 2.2 엔티티 — `Event extends BaseEntity`

- `@Entity @Table(name = "events")`, `@Id @GeneratedValue(IDENTITY)`.
- 필드: title(String), description(String, TEXT), location(String), startAt(LocalDateTime), endAt(LocalDateTime), allDay(boolean).
- `BaseEntity`에서 createdAt/updatedAt/createdBy/updatedBy/deletedAt/version/`isDeleted()`/`softDelete()` 상속.
- 생성/수정 메서드(notice 관례):
  - `static Event create(title, description, location, startAt, endAt, allDay)` — 팩토리.
  - `update(...)` — PUT 전체 교체(감사필드 제외 전 필드 덮어쓰기).
  - `applyPatch(title, description, location, startAt, endAt, Boolean allDay)` — PATCH 부분 갱신(null 인자는 미변경).
    > **PATCH의 null=미변경 관례 한계**: end_at을 명시적으로 비우려면(있던 종료시각 제거) PUT을 사용한다. notice가 isPinned에서 택한 것과 동일한 단순화이며, end_at 클리어는 드문 케이스라 PUT으로 충분.
- `created_by`/`updated_by`는 `AuditorAware`(SecurityContext)가 자동 주입 — 서비스가 수동 세팅하지 않는다.

## 3. 리포지토리 — `EventRepository extends JpaRepository<Event, Long>, JpaSpecificationExecutor<Event>`

- `Optional<Event> findByIdAndDeletedAtIsNull(Long id)` — 상세·수정·삭제 공용.
- `List<EventRefRow> findReferencesByMedia(String pattern)` — 네이티브
  `select id as id, title as title from events where deleted_at is null and description ~ :pattern` (id·title 프로젝션).
- **`incrementViewCount` 없음**(view_count 미보유).
- 동적 필터는 `EventSpecifications`(notice `NoticeSpecifications` 패턴 답습).

### 3.1 `EventSpecifications.filter(DateRange range, List<Long> taggedIds)`

`null` 인자는 술어에서 제외. 항상 `deletedAt IS NULL` 기본 술어.

- **range**(겹침) — `range`가 있으면(from·toExclusive 보유):
  `start_at < toExclusive` **AND** `COALESCE(end_at, start_at) >= from`.
  - Criteria로: `cb.lessThan(root.get("startAt"), toExclusive)`,
    `cb.greaterThanOrEqualTo(cb.coalesce(root.get("endAt"), root.get("startAt")), from)`.
  - `range`가 null이면(파라미터 미지정) 범위 술어 제외 → 전체(페이지네이션만).
- **taggedIds** — **서비스가 미리 해석해 넘긴 id 목록**(`List<Long>`)을 `id IN (...)`로. `null`이면 술어 제외,
  빈 리스트면 결과 없음(불가능 술어). Specification은 순수 조건 빌더로 유지하고 태그 id 해석은 서비스 책임(notice와 동일).

> **`DateRange`는 서비스/컨트롤러 경계에서 해석한 `(from, toExclusive)` 값 객체**다(반열림 구간). 파라미터 → 구간 변환과 검증(쌍 누락·역전)은 컨트롤러/서비스가 맡고, Specification은 이미 정규화된 구간만 받는다.

## 4. 미디어 참조 — `EventReferenceProvider implements MediaReferenceProvider` (`@Component`)

```text
findReferences(long mediaId):
    pattern = "media:" + mediaId + "($|[^0-9])"      // 경계 안전 — 42가 420/421에 매칭 안 됨
    rows    = repo.findReferencesByMedia(pattern)
    return rows.map(r -> new ContentRef("event", r.id(), r.title()))
```

- **`ContentRef.type`은 소문자 `"event"`** — 미디어 참조 API(`GET /references`, `MEDIA_IN_USE` 봉투)가 노출하는
  계약 값(스펙 §5.10 UNION의 `'event'`). 매칭 대상은 `description` 컬럼.
- soft-deleted 이벤트는 쿼리에서 제외 → 삭제된 이벤트가 미디어 차단형 삭제를 막지 않는다(자기 치유).
- MediaService가 모든 `MediaReferenceProvider`를 주입받아 합집합을 구한다 — media는 event를 모른다(content → media 단방향).

## 5. 서비스 — `EventService` (`@Transactional(readOnly=true)`)

의존: `EventRepository`, `ContentTagService`(둘 다 기존 재사용). **`AuthorDisplayService` 의존 없음**.

- **list**(DateRange range, Long tagId, Pageable) → `Page<EventCardResponse>`
  - `tagId`가 있으면 **먼저** `ContentTagService.resourceIdsWithTag(EVENT, tagId)`로 `taggedIds`를 해석.
  - `repo.findAll(EventSpecifications.filter(range, taggedIds), pageable)`.
  - 결과 id 모아 `getTagsByResources(EVENT, ids)` **배치 조회**(작성자 배치 조회 없음).
  - `description` 제외 카드로 매핑. 기본 정렬 `start_at ASC`(컨트롤러 `@PageableDefault`).
- **get**(id) → `EventDetailResponse`
  - `findByIdAndDeletedAtIsNull(id)`(없으면 404 `RESOURCE_NOT_FOUND`) → `detail(event)`.
  - **조회수 증가 없음** → 읽기 경로가 순수 read-only(notice/sermon과 다른 점). `@Transactional` 불필요(클래스 기본 readOnly로 충분).
- **create**(req) `@Transactional` → detail
  - `repo.save(Event.create(...))`(created_by/updated_by 자동) → `replaceLinks(EVENT, id, req.tagIds())`.
- **update**(id, req) `@Transactional` — PUT 전체 교체
  - load(404) → **명시적 version 비교**(`req.version != e.version` → `OPTIMISTIC_LOCK_CONFLICT`) → `e.update(...)` →
    `replaceLinks(EVENT, id, tagIds)`(null이면 전체 해제) → **`repository.flush()`** → `detail(e)`.
- **patch**(id, req) `@Transactional` — 부분 갱신
  - load(404) → version 비교 → `e.applyPatch(...)`(전달 필드만) →
    tagIds **제공 시에만** `replaceLinks`(null이면 태그 유지) → **`repository.flush()`** → `detail(e)`.
  - **flush 이유**: tagIds 미제공 PATCH는 detail이 `content_tags`만 쿼리해 dirty event가 auto-flush 되지 않는다.
    mutation 직후 명시적 flush로 버전 UPDATE를 강제(in-memory `version` N→N+1)해야 응답 `version`이 post-increment 값이 되어
    클라이언트의 다음 수정 재전송이 stale 409를 맞지 않는다(notice §5와 동일 정합).
- **delete**(id) `@Transactional`
  - load(404) → `e.softDelete()` → `ContentTagService.cleanUp(EVENT, id)`.

### 5.1 교차 필드 검증 — `end_at >= start_at`

- **create/update**: 요청 record에 `@AssertTrue` 메서드로 `endAt == null || startAt == null || !endAt.isBefore(startAt)` 검증(둘 다 record에 존재).
- **patch**: start_at/end_at 중 일부만 올 수 있어 record 단독 검증 불가 → 서비스가 patch 적용 **후** effective start/end를 비교,
  역전이면 `INVALID_INPUT_VALUE`. (DB 값과 합쳐야 판정 가능하므로 서비스 책임.)

> 낙관락 이중 안전망: 서비스의 명시적 version 비교가 "stale 클라이언트 버전"을 결정적으로 409로 만든다.
> load~flush 사이 동시 커밋은 JPA `@Version`이 `ObjectOptimisticLockingFailureException`을 던지고,
> 이는 `GlobalExceptionHandler`가 **이미** `OPTIMISTIC_LOCK_CONFLICT`로 매핑한다(추가 작업 불필요).

## 6. API 계약

경로 인가는 기존 `SecurityConfig`로 충족(변경 불필요): `GET /api/events/**` 공개,
`/api/admin/**`는 `authenticated()` + 메서드 `@PreAuthorize`. `EVENT_WRITE` 권한은 V2에 **이미 시드됨**.

### 6.1 `EventController` (공개)

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/api/events` | 목록. `?year=&month=` 또는 `?startDate=&endDate=`(범위, 겹침) + `tagId` + `?page&size&sort`. `Page<EventCardResponse>`(description 제외). 기본 정렬 `start_at ASC`. |
| GET | `/api/events/{id}` | 상세. `EventDetailResponse`(조회수 증가 없음). |

**범위 파라미터 해석(컨트롤러):**
- `year`(int) + `month`(int 1–12) **둘 다** 제공 → 그 달: `from = 해당월 1일 00:00`, `toExclusive = 다음달 1일 00:00`.
- 아니면 `startDate`(LocalDate) + `endDate`(LocalDate) **둘 다** 제공 → `from = startDate 00:00`, `toExclusive = endDate.plusDays(1) 00:00`.
- 둘 다 없으면 → 범위 없음(전체, 페이지네이션만).
- **검증 오류 → `400 INVALID_INPUT_VALUE`**: 한 쌍 중 하나만 제공(`year`만/`startDate`만 등), `month` 범위 밖(1–12 아님), `endDate < startDate`.
- year/month 와 startDate/endDate가 **동시 제공**되면 year/month 우선(명시적 우선순위, 문서화).

> **목록 JSON 계약**: 컨트롤러 반환 타입은 Java `Page<EventCardResponse>`지만, `WebConfig`의
> `@EnableSpringDataWebSupport(pageSerializationMode = VIA_DTO)`가 표준 envelope
> `{ "content": [...], "page": { "size", "number", "totalElements", "totalPages" } }`로 직렬화한다(스펙 §5 목록 규약, notice·sermon과 동일).
> `EventApiTest`가 `$.page.totalElements`·`$.content[*]`로 검증한다.

> **달력 한 달 분량과 페이지 size**: 기본 size는 표준 10을 유지한다(notice·sermon과 일관). 달력 뷰는 프론트가 `?size=` 를 키워 호출한다
> (스펙 §5 "size로 개수 조절" 철학). 백엔드는 별도 비페이지 엔드포인트를 두지 않는다.

### 6.2 `AdminEventController` (`@PreAuthorize("hasAuthority('EVENT_WRITE')")`)

| 메서드 | 경로 | 응답 |
|---|---|---|
| POST | `/api/admin/events` | 201 + detail |
| PUT | `/api/admin/events/{id}` | 200 + detail (전체 수정) |
| PATCH | `/api/admin/events/{id}` | 200 + detail (부분 수정) |
| DELETE | `/api/admin/events/{id}` | 204 |

### 6.3 DTO

- `EventCreateRequest` — `title`(@NotBlank, @Size(max=200)), `description`(@Size(max=50000) — TEXT지만 스펙 §5 최소검증 상한 1개),
  `location`(@Size(max=200)), `startAt`(@NotNull LocalDateTime), `endAt`(LocalDateTime, nullable), `allDay`(Boolean, 미지정 시 false),
  `tagIds`(List<Long>). `@AssertTrue`로 `endAt >= startAt` 교차검증.
- `EventUpdateRequest` — create 필드 + `version`(@NotNull). PUT 전체 교체. 동일 `@AssertTrue` 교차검증.
- `EventPatchRequest` — title(@Size), description(@Size), location(@Size), startAt, endAt, allDay 전부 nullable + `version`(@NotNull).
  전달된 필드만 적용. start/end 교차검증은 서비스(§5.1).
- `EventCardResponse` — id, title, location, startAt, endAt, allDay, tags(`List<TagResponse>`). **description·author·viewCount 없음**.
- `EventDetailResponse` — id, title, description, location, startAt, endAt, allDay, createdAt, updatedAt,
  **version**(편집 재전송용 — update/patch 응답은 §5 flush로 post-increment 값 보장), tags. **author 없음**.

  > 인덱스 회귀 검증은 기존 **`MigrationIndexTest`**(Flyway-on + `pg_indexes`)에 `idx_events_start_at` 부분 조건(`WHERE deleted_at IS NULL`) 확인을 **추가**한다.
- `EventReferenceProviderTest` — 경계 안전 매칭(`42`가 `media:420` 미매칭), soft-deleted 제외, 다건 합산, type `"event"`.
- `EventServiceTest` — create/update/patch/delete, **version 충돌 → 409**, 태그 연결/정리, flush 후 version N+1,
  **PATCH(tagIds 미제공) 응답 version이 N+1로 증가하고 그 version으로 즉시 다음 수정 성공**(stale 409 회피 회귀 가드),
  **patch 후 start/end 역전 → 400**(서비스 교차검증).
- `EventApiTest`(Testcontainers E2E) — 공개 목록(`year/month`·`startDate/endDate`, **여러 날 행사 겹침**, tag 필터, `start_at ASC` 정렬),

  검증 오류(쌍 누락·`month` 범위 밖·`endDate<startDate`·`endAt<startAt`), **수정 응답 version으로 연속 PATCH 성공**.

## 9. 비목표 / 범위 밖

- **반복 일정**(recurrence) — 스펙 §5.6 명시적으로 1차 범위 제외. 단건만.
- **작성자 표시** — §1 결정대로 미노출(엔티티 감사 컬럼은 보유).
- **view_count / 조회수** — Event는 카운팅 안 함(Redis §9 대상 아님).
- **`?q=` 전문/제목 검색** — §5.6 미요구. range + tag 필터만.
- **`/api/main` 통합**(스펙 §5.9 "다가오는 일정") — 별도 Common 이슈. 본 도메인의 range 조회(`startDate=오늘`)로 이미 커버됨.
- 멀티테넌시 컬럼(스펙 §2) — 절대 추가 안 함.
- 본문 HTML 변환·새니타이즈 — 프론트 책임. 서버는 마크다운 원본만 저장(최소 길이 검증만).
- 미디어 바이너리 저장 — event는 본문에 `media:{id}` 문자열만 둔다(별도 연결 테이블 없음).
