# 공지(Notice) 도메인 설계 — 이슈 #13 (D8)

스펙 §5.7 구현. 교회 공지를 등록·조회·관리하는 단일 교회용 도메인을 추가한다.
본문은 마크다운 원본을 저장하고 렌더링은 프론트 책임이다(스펙 §5 본문 저장 정책).

## 0. 배경 — Sermon(D7) 패턴의 답습

Notice는 Sermon(D7)이 세운 패턴을 그대로 따르는 **더 단순한 형제 도메인**이다. 새로운 아키텍처
결정은 없고, 기존 자산을 그대로 재사용한다.

기존 자산 재사용(신규 컴포넌트 0개):
- `MediaReferenceProvider`(media 도메인 SPI) — 본문 `media:{id}` 참조 추적. sermon에 이은 두 번째 구현체.
- `ContentTagService`(tag 도메인) — `ContentResourceType.NOTICE`는 이미 enum에 존재. 그대로 연결.
- `AuthorDisplayService`(member 도메인) — `updated_by` → 표시 이름 해석. sermon이 들인 재사용 컴포넌트를 그대로 상속.
- `BaseEntity`(global/common) — 감사/소프트삭제/낙관락 컬럼 상속.

## 1. 확정된 설계 결정 (브레인스토밍 합의)

| 항목 | 결정 | 근거 |
|---|---|---|
| **상단고정 정렬** | **기본 정렬만** — `@PageableDefault(sort={isPinned,createdAt}, DESC)`. 클라이언트 `?sort=` 로 덮어쓰기 허용 | 스펙 §5.7 "기본 정렬: is_pinned DESC, created_at DESC(고정 먼저, 그 다음 최신)" — 강제 prefix 요구 문구 없음. sermon `@PageableDefault` 선례와 동일. PG/JPA에서 boolean DESC는 true(고정) 먼저. |
| **검색 `?q=`** | `title` 에 `lower() LIKE` (본문 `content` 제외) | 공지 카드 메타 텍스트가 `title` 뿐. sermon `?q=`(카드 메타 필드) 패턴 일관. 인덱스 없는 TEXT 풀스캔 회피. |
| **조회수 증가** | 상세 GET마다 **원자적 `UPDATE view_count = view_count+1`** (별도 쿼리, `@Version` 우회) | sermon과 동일. 읽기 경로의 쓰기가 낙관락과 충돌하지 않아야 함. |
| **작성자 표시** | 재사용 컴포넌트 `AuthorDisplayService` | sermon이 들인 도메인 횡단 재사용 그대로 상속(탈퇴 마스킹·N+1 회피 단일 보장). |
| **미디어 참조 매칭** | **경계 안전**(PG 정규식 `media:{id}($\|[^0-9])`) | 순진한 `LIKE '%media:42%'`는 `media:420`도 매칭 → 잘못된 삭제 차단. sermon 선례 그대로. |
| **필드 셋** | title, content, is_pinned, view_count (+ BaseEntity 감사 컬럼) | 스펙 §5.7. sermon 대비 preacher/series/scripture/dates/media URL 없음 — 더 단순. |
| **수정 응답 version 정합성** (리뷰 반영) | update/patch는 mutation 후 **`repository.flush()`로 버전 UPDATE를 즉시 반영**한 뒤 detail 생성 | `@Version` 증가는 flush 시점. detail의 유일한 쿼리는 `content_tags`(다른 테이블)라 Hibernate 선택적 auto-flush가 dirty notice를 flush하지 않음 → flush 없이는 응답 `version`이 pre-increment(stale)로 내려가 클라이언트의 다음 수정이 spurious 409. flush로 in-memory version N→N+1 보장. (sermon은 현재 이 flush가 없어 동일 잠재 이슈 — 별도 후속.) |

## 2. 데이터 모델

### 2.1 마이그레이션 — `V8__create_notices.sql`

> V7은 `V7__create_sermons.sql`이 점유 중 → notice는 **V8**.

```sql
-- 공지 콘텐츠(스펙 §5.7). BaseEntity 상속, 감사/소프트삭제/낙관락 컬럼은 V7(sermons)에서 확정한 관례를 따른다.
-- 본문 content는 마크다운 원본(TEXT), 본문 내 이미지는 media:{id}로 참조(스펙 §5 본문 정책).
CREATE TABLE notices (
    id          BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title       VARCHAR(200) NOT NULL,
    content     TEXT,                                  -- 마크다운 원본, media:{id} 참조
    is_pinned   BOOLEAN      NOT NULL DEFAULT FALSE,
    view_count  BIGINT       NOT NULL DEFAULT 0,
    created_at  TIMESTAMP    NOT NULL,                 -- BaseEntity @CreatedDate
    updated_at  TIMESTAMP,                             -- BaseEntity @LastModifiedDate
    created_by  BIGINT       REFERENCES members (id),  -- 최초 작성자(audit)
    updated_by  BIGINT       REFERENCES members (id),  -- 마지막 수정자 = 표시용 작성자
    deleted_at  TIMESTAMP,                             -- soft delete
    version     BIGINT       NOT NULL DEFAULT 0        -- 낙관적 락
);

-- 목록 기본 정렬 = is_pinned DESC, created_at DESC, 미삭제만(스펙 §6 부분 인덱스).
CREATE INDEX idx_notices_pinned_created ON notices (is_pinned DESC, created_at DESC) WHERE deleted_at IS NULL;
```

감사 컬럼 타입은 V3(members)·V7(sermons)의 관례(`TIMESTAMP`, `BIGINT`)를 따른다. `ddl-auto: validate`이므로
엔티티 매핑과 정확히 일치해야 한다(빌드로 검증).

### 2.2 엔티티 — `Notice extends BaseEntity`

- `@Entity @Table(name = "notices")`, `@Id @GeneratedValue(IDENTITY)`.
- 필드: title, content, isPinned(boolean), viewCount(Long).
- `BaseEntity`에서 createdAt/updatedAt/createdBy/updatedBy/deletedAt/version/`isDeleted()`/`softDelete()` 상속.
- 생성/수정 메서드(sermon 관례):
  - `static Notice create(...)` — 팩토리. `viewCount=0`, `isPinned`는 요청값(미지정 시 false).
  - `update(...)` — PUT 전체 교체.
  - `applyPatch(...)` — PATCH 부분 갱신(전달된 필드만, **`isPinned` 상단고정 토글 포함**).
- **`viewCount`에는 앱 코드용 세터가 없다** — 오직 리포지토리 원자 쿼리로만 증가(@Version 미증가).
- `created_by`/`updated_by`는 `AuditorAware`(SecurityContext)가 자동 주입 — 서비스가 수동 세팅하지 않는다.

## 3. 리포지토리 — `NoticeRepository extends JpaRepository<Notice, Long>, JpaSpecificationExecutor<Notice>`

- `Optional<Notice> findByIdAndDeletedAtIsNull(Long id)` — 상세·수정·삭제 공용.
- `@Modifying(clearAutomatically = true, flushAutomatically = true)`
  `int incrementViewCount(Long id)` — `update Notice n set n.viewCount = n.viewCount + 1 where n.id = :id and n.deletedAt is null`.
- `List<NoticeRefRow> findReferencesByMedia(String pattern)` — 네이티브
  `select id as id, title as title from notices where deleted_at is null and content ~ :pattern` (id·title 프로젝션).
- 동적 필터는 `NoticeSpecifications`(sermon `SermonSpecifications` 패턴 답습).

### 3.1 `NoticeSpecifications.filter(q, taggedIds)`

`null` 인자는 술어에서 제외. 항상 `deletedAt IS NULL` 기본 술어.

- `q` — `lower(title) LIKE %q%`(공지 카드 메타 텍스트가 title뿐이라 단일 필드).
- `taggedIds` — **서비스가 미리 해석해 넘긴 id 목록**(`List<Long>`)을 `id IN (...)`로. `null`이면 술어 제외,
  빈 리스트면 결과 없음(불가능 술어). Specification은 순수 조건 빌더로 유지하고 태그 id 해석은 서비스 책임.

## 4. 미디어 참조 — `NoticeReferenceProvider implements MediaReferenceProvider` (`@Component`)

```text
findReferences(long mediaId):
    pattern = "media:" + mediaId + "($|[^0-9])"      // 경계 안전 — 42가 420/421에 매칭 안 됨
    rows    = repo.findReferencesByMedia(pattern)
    return rows.map(r -> new ContentRef("notice", r.id(), r.title()))
```

- **`ContentRef.type`은 소문자 `"notice"`** — 미디어 참조 API(`GET /references`, `MEDIA_IN_USE` 봉투)가 노출하는
  계약 값(스펙 §5.10 예시 `"notice"`/`"sermon"`).
- `media:`는 id 바로 앞 고정 경계라 후행 경계만 검사하면 충분.
- soft-deleted 공지는 쿼리에서 제외 → 삭제된 공지가 미디어 차단형 삭제를 막지 않는다(자기 치유).
- MediaService가 모든 `MediaReferenceProvider`를 주입받아 합집합을 구한다 — media는 notice를 모른다(content → media 단방향).

## 5. 서비스 — `NoticeService` (`@Transactional(readOnly=true)`)

의존: `NoticeRepository`, `ContentTagService`, `AuthorDisplayService`(셋 다 기존 재사용).

- **list**(q, tagId, pageable) → `Page<NoticeCardResponse>`
  - `tagId`가 있으면 **먼저** `ContentTagService.resourceIdsWithTag(NOTICE, tagId)`로 `taggedIds`를 해석.
  - `repo.findAll(NoticeSpecifications.filter(q, taggedIds), pageable)`.
  - 결과 id 모아 `getTagsByResources(NOTICE, ids)` + `displayNames(updatedByIds)` **배치 조회**.
  - `content` 제외 카드로 매핑. 기본 정렬 `is_pinned DESC, created_at DESC`(컨트롤러 `@PageableDefault`).
- **get**(id) `@Transactional` → `NoticeDetailResponse`
  - `incrementViewCount(id)` → `findByIdAndDeletedAtIsNull(id)`(없으면 404 `RESOURCE_NOT_FOUND`).
  - 증가 후 로드 → 반환 view_count에 +1 반영. 태그·작성자 결합. `content`·`version` 포함.
  - **의도된 trade-off(리뷰 반영)**: 존재 확인보다 조회수 UPDATE를 먼저 실행한다. 없는/soft-deleted id면
    `UPDATE ... WHERE deleted_at IS NULL`이 0-row(무해)로 끝나고 이어진 SELECT가 비어 404가 된다. 사전 존재
    확인 라운드트립을 1회 아낀다(sermon과 일관). 동시 삭제 레이스에서 "증가 후 404"는 의미상 무시 가능
    (어차피 삭제될 행의 카운트). 더 엄격한 계약이 필요하면 `incrementViewCount` 반환 0을 바로 404로 매핑하는
    선택지가 있으나, 본 설계는 sermon 일관성을 택해 현 동작을 유지한다.
- **create**(req) `@Transactional` → detail
  - `repo.save(Notice.create(...))`(created_by/updated_by 자동) → `replaceLinks(NOTICE, id, req.tagIds())`.
- **update**(id, req) `@Transactional` — PUT 전체 교체
  - load(404) → **명시적 version 비교**(`req.version != n.version` → `OPTIMISTIC_LOCK_CONFLICT`) → `n.update(...)` →
    `replaceLinks(NOTICE, id, tagIds)`(null이면 전체 해제) → **`repository.flush()`** → `detail(n)`.
- **patch**(id, req) `@Transactional` — 부분 갱신
  - load(404) → version 비교 → `n.applyPatch(...)`(전달 필드만, `isPinned` 토글 포함) →
    tagIds **제공 시에만** `replaceLinks`(null이면 태그 유지) → **`repository.flush()`** → `detail(n)`.
  - **flush 이유(리뷰 반영)**: tagIds 미제공 PATCH는 detail이 `content_tags`만 쿼리해 dirty notice가 auto-flush
    되지 않는다. mutation 직후 명시적 flush로 버전 UPDATE를 강제(in-memory `version` N→N+1)해야 응답 `version`이
    post-increment 값이 되어 클라이언트의 다음 수정 재전송이 stale 409를 맞지 않는다.
- **delete**(id) `@Transactional`
  - load(404) → `n.softDelete()` → `ContentTagService.cleanUp(NOTICE, id)`.

> 낙관락 이중 안전망: 서비스의 명시적 version 비교가 "stale 클라이언트 버전"을 결정적으로 409로 만든다.
> load~flush 사이 동시 커밋은 JPA `@Version`이 `ObjectOptimisticLockingFailureException`을 던지고,
> 이는 `GlobalExceptionHandler`가 **이미** `OPTIMISTIC_LOCK_CONFLICT`로 매핑한다(추가 작업 불필요).

## 6. API 계약

경로 인가는 기존 `SecurityConfig`로 충족(변경 불필요): `GET /api/notices/**` 공개,
`/api/admin/**`는 `authenticated()` + 메서드 `@PreAuthorize`. `NOTICE_WRITE` 권한은 V2에 이미 시드됨.

### 6.1 `NoticeController` (공개)

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/api/notices` | 목록. `?page&size&sort=isPinned,desc&...` + `q`·`tagId`. `Page<NoticeCardResponse>`(content 제외). 기본 정렬 `isPinned DESC, createdAt DESC`. |
| GET | `/api/notices/{id}` | 상세. 조회수 +1. `NoticeDetailResponse`. |

### 6.2 `AdminNoticeController` (`@PreAuthorize("hasAuthority('NOTICE_WRITE')")`)

| 메서드 | 경로 | 응답 |
|---|---|---|
| POST | `/api/admin/notices` | 201 + detail |
| PUT | `/api/admin/notices/{id}` | 200 + detail (전체 수정) |
| PATCH | `/api/admin/notices/{id}` | 200 + detail (부분 수정, 상단고정 토글 등) |
| DELETE | `/api/admin/notices/{id}` | 204 |

### 6.3 DTO

- `NoticeCreateRequest` — DB 컬럼 길이와 일치하는 `@Size(max=...)` 명시:
  `title`(@NotBlank, @Size(max=200)), `content`(TEXT — 스펙 §5 최소 검증으로 합리적 길이 상한 1개 부여),
  `isPinned`(Boolean, 미지정 시 false), `tagIds`(List<Long>).
- `NoticeUpdateRequest` — create 필드 + `version`(@NotNull). PUT 전체 교체.
- `NoticePatchRequest` — 전부 nullable + `version`(@NotNull). 전달된 필드만 적용(`isPinned` 토글 포함).
- `NoticeCardResponse` — id, title, isPinned, viewCount, createdAt, tags(`List<TagResponse>`), author(String). **content 없음**.
- `NoticeDetailResponse` — 카드 전체 + content, updatedAt, **version**(편집 재전송용 — update/patch 응답은 §5 flush로 post-increment 값 보장), tags, author.

## 9. 비목표 / 범위 밖

- 멀티테넌시 컬럼(스펙 §2) — 절대 추가 안 함.
- 본문 HTML 변환·새니타이즈 — 프론트 책임. 서버는 마크다운 원본만 저장(최소 길이 검증만).
- 미디어 바이너리 저장 — notice는 본문에 `media:{id}` 문자열만 둔다(별도 연결 테이블 없음).
- 전문 검색(PG full-text) — `content` 검색은 범위 밖(카드 필드 `title` `LIKE`만).
- 상단고정 강제 prefix 정렬 — 스펙은 "기본 정렬"만 요구. 클라이언트 `?sort=` 덮어쓰기 허용(sermon 선례).
