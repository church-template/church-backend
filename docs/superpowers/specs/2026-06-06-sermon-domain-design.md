# 설교(Sermon) 도메인 설계 — 이슈 #12 (D7)

스펙 §5.5 구현. 설교 콘텐츠를 등록·조회·관리하는 단일 교회용 도메인을 추가한다.
본문은 마크다운 원본을 저장하고 렌더링은 프론트 책임이다(스펙 §5 본문 저장 정책).

## 0. 배경 — 기존 자산의 첫 사용처(3) + 신규 컴포넌트(1)

Sermon은 기존에 깔려 있던 자산 3종의 **첫 사용처**이자, 재사용 컴포넌트 1종을 새로 들인다.
이 도메인이 세우는 패턴을 이후 notice/event/department/gallery/bulletin(D8~D11)이 그대로 답습한다.

기존 자산의 첫 사용처:
- `MediaReferenceProvider`(media 도메인 SPI, 현재 구현체 0개) — 본문 `media:{id}` 참조 추적의 **첫 구현**.
- `ContentTagService`(tag 도메인) — `ContentResourceType.SERMON`은 이미 enum에 존재. 그대로 연결.
- `BaseEntity`(global/common) — `extends BaseEntity`하는 **첫 엔티티**. V7 마이그레이션이 감사 컬럼 타입을 처음으로 확정한다(ddl-auto: validate).

신규:
- `AuthorDisplayService`(member 도메인) — `updated_by` → 표시 이름 해석의 첫 도메인. 이후 4개 도메인이 상속.

## 1. 확정된 설계 결정 (브레인스토밍 합의)

| 항목 | 결정 | 근거 |
|---|---|---|
| **조회수 증가** | 상세 GET마다 **원자적 `UPDATE view_count = view_count+1`** (별도 쿼리, `@Version` 우회) | 읽기 경로의 쓰기가 낙관락과 충돌하지 않아야 함. 엔티티 로드/세이브와 무관한 1줄 쿼리. notice 등 재사용. |
| **검색 `?q=`** | `title + preacher + series + scripture` 에 `lower() LIKE` (본문 `content` 제외) | 카드 메타 필드만 검색. 인덱스 없는 TEXT 풀스캔 회피. 교회 규모에 충분. |
| **작성자 표시** | member 도메인에 **재사용 컴포넌트** `AuthorDisplayService` | `ContentTagService`와 동일한 도메인 횡단 재사용 패턴. 탈퇴 마스킹·N+1 회피를 한 곳에서 보장하고 이후 4개 도메인이 상속. |
| **미디어 참조 매칭** | **경계 안전**(PG 정규식 `media:{id}($|[^0-9])`) | 순진한 `LIKE '%media:42%'`는 `media:420`도 매칭 → 잘못된 삭제 차단. sermon이 첫 구현이라 전 도메인 선례. |
| **필드 셋** | **스펙 §5.5 전체 셋**(이슈 본문 그대로) | series·audio_url 포함, content는 마크다운 본문. |

## 2. 데이터 모델

### 2.1 마이그레이션 — `V7__create_sermons.sql`

> V6은 `V6__media_stored_path_unique.sql`(media.stored_path 유니크 제약)이 이미 점유 중 → sermon은 **V7**.

```sql
-- 설교 콘텐츠(스펙 §5.5). BaseEntity를 상속하는 첫 엔티티 — 감사/소프트삭제/낙관락 컬럼을 여기서 처음 확정.
-- 본문 content는 마크다운 원본(TEXT), 본문 내 이미지는 media:{id}로 참조(스펙 §5 본문 정책).
CREATE TABLE sermons (
    id          BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title       VARCHAR(200) NOT NULL,
    preacher    VARCHAR(100) NOT NULL,
    series      VARCHAR(100),
    scripture   VARCHAR(200),
    content     TEXT,                                  -- 마크다운 원본, media:{id} 참조
    video_url   VARCHAR(500),
    audio_url   VARCHAR(500),
    preached_at DATE         NOT NULL,
    view_count  BIGINT       NOT NULL DEFAULT 0,
    created_at  TIMESTAMP    NOT NULL,                 -- BaseTimeEntity @CreatedDate
    updated_at  TIMESTAMP,                             -- BaseEntity @LastModifiedDate
    created_by  BIGINT       REFERENCES members (id),  -- 최초 작성자(audit)
    updated_by  BIGINT       REFERENCES members (id),  -- 마지막 수정자 = 표시용 작성자
    deleted_at  TIMESTAMP,                             -- soft delete
    version     BIGINT       NOT NULL DEFAULT 0        -- 낙관적 락
);

-- 목록 기본 정렬 = preached_at DESC, 미삭제만(스펙 §6 부분 인덱스).
CREATE INDEX idx_sermons_preached_at ON sermons (preached_at DESC) WHERE deleted_at IS NULL;
```

감사 컬럼 타입은 V3(members)의 관례(`TIMESTAMP`, `BIGINT`)를 따른다. `ddl-auto: validate`이므로
엔티티 매핑과 정확히 일치해야 한다(빌드로 검증).

### 2.2 엔티티 — `Sermon extends BaseEntity`

- `@Entity @Table(name = "sermons")`, `@Id @GeneratedValue(IDENTITY)`.
- 필드: title, preacher, series, scripture, content, videoUrl, audioUrl, preachedAt(LocalDate), viewCount(Long).
- `BaseEntity`에서 createdAt/updatedAt/createdBy/updatedBy/deletedAt/version/`isDeleted()`/`softDelete()` 상속.
- 생성/수정 메서드(기존 엔티티 관례):
  - `static Sermon create(...)` — 팩토리. `viewCount=0`.
  - `update(...)` — PUT 전체 교체.
  - `applyPatch(...)` — PATCH 부분 갱신(전달된 필드만).
- **`viewCount`에는 앱 코드용 세터가 없다** — 오직 리포지토리 원자 쿼리로만 증가(@Version 미증가).
- `created_by`/`updated_by`는 `AuditorAware`(SecurityContext)가 자동 주입 — 서비스가 수동 세팅하지 않는다.

## 3. 리포지토리 — `SermonRepository extends JpaRepository<Sermon, Long>, JpaSpecificationExecutor<Sermon>`

- `Optional<Sermon> findByIdAndDeletedAtIsNull(Long id)` — 상세·수정·삭제 공용.
- `@Modifying(clearAutomatically = true, flushAutomatically = true)`
  `int incrementViewCount(Long id)` — `UPDATE Sermon s SET s.viewCount = s.viewCount + 1 WHERE s.id = :id AND s.deletedAt IS NULL`.
- `List<SermonRefRow> findReferencesByMedia(String pattern)` — 네이티브 쿼리
  `SELECT id, title FROM sermons WHERE deleted_at IS NULL AND content ~ :pattern` (id·title 프로젝션).
- 동적 필터는 `SermonSpecifications`(media `MediaSpecifications` 패턴 답습).

### 3.1 `SermonSpecifications.filter(...)`

`null` 인자는 술어에서 제외. 항상 `deletedAt IS NULL` 기본 술어.

- `preacher` — 일치(eq).
- `series` — 일치(eq).
- `from`/`to` — `preachedAt >= from`, `preachedAt < to.plusDays(1)`(해당 날짜 포함, 상한 배타).
- `q` — `lower(title|preacher|series|scripture) LIKE %q%`의 OR 묶음.
- `taggedIds` — **서비스가 미리 해석해 넘긴 id 목록**(`List<Long>`)을 `id IN (...)`로. `null`이면 술어 제외, 빈 리스트면 결과 없음(불가능 술어). Specification은 **순수 조건 빌더로 유지**하고 태그 id 해석은 서비스 책임 — `MediaSpecifications`와 동일 원칙(구현 책임 비혼합).

## 4. 미디어 참조 — `SermonReferenceProvider implements MediaReferenceProvider` (`@Component`)

```text
findReferences(long mediaId):
    pattern = "media:" + mediaId + "($|[^0-9])"      // 경계 안전 — 42가 420/421에 매칭 안 됨
    rows    = repo.findReferencesByMedia(pattern)
    return rows.map(r -> new ContentRef("sermon", r.id(), r.title()))
```

- **`ContentRef.type`은 소문자 `"sermon"`** — 미디어 참조 API(`GET /references`, `MEDIA_IN_USE` 봉투)가 노출하는 계약 값이라 기존 관례를 따른다(스펙 §5.10 예시 `"notice"`/`"sermon"`, `MediaServiceTest`).
- `media:`는 id 바로 앞 고정 경계라 **후행 경계만** 검사하면 충분(`media:142`에 `media:42`는 substring 아님).
- soft-deleted 설교는 쿼리에서 제외 → 삭제된 설교가 미디어 차단형 삭제를 막지 않는다(자기 치유).
- MediaService가 모든 `MediaReferenceProvider`를 주입받아 합집합을 구한다 — media는 sermon을 모른다(content → media 단방향, ArchUnit `global ↛ domain`과 무관, 도메인↔도메인은 허용).

## 5. 작성자 표시 — `AuthorDisplayService` (신규, member 도메인, `@Service @Transactional(readOnly=true)`)

`ContentTagService`와 동일한 "도메인 횡단 재사용 컴포넌트". 모든 메서드 public(타 패키지 호출).

- `String displayName(Long memberId)` — 단건(상세 응답용).
- `Map<Long, String> displayNames(Collection<Long> memberIds)` — 배치 맵(목록 N+1 회피, 요청한 모든 id를 키로 갖는 완전한 맵).
- 해석 규칙:
  - 회원 존재 & 미삭제 → `name`.
  - 회원 **soft-deleted → "(탈퇴한 사용자)"** (FK 유지, 이름만 마스킹).
  - id == null → `"(알 수 없음)"`.
- **soft-deleted 회원도 조회**해야 마스킹이 되므로, `MemberRepository`에 미삭제 필터 없는 프로젝션 추가:
  `SELECT m.id, m.name, m.deletedAt FROM Member m WHERE m.id IN :ids` (id/name/deletedAt 3필드 뷰).

## 6. 서비스 — `SermonService` (`@Transactional(readOnly=true)`)

의존: `SermonRepository`, `ContentTagService`, `AuthorDisplayService`.

- **list**(filter, pageable) → `Page<SermonCardResponse>`
  - `tagId`가 있으면 **먼저** `ContentTagService.resourceIdsWithTag(SERMON, tagId)`로 `taggedIds`를 해석.
  - `repo.findAll(SermonSpecifications.filter(preacher, series, from, to, q, taggedIds), pageable)`.
  - 결과 id 모아 `getTagsByResources(SERMON, ids)` + `displayNames(updatedByIds)` **배치 조회**.
  - `content` 제외 카드로 매핑. 기본 정렬 `preached_at DESC`(컨트롤러 `@PageableDefault`).
- **get**(id) `@Transactional` → `SermonDetailResponse`
  - `incrementViewCount(id)` → `findByIdAndDeletedAtIsNull(id)`(없으면 404 `RESOURCE_NOT_FOUND`).
  - 증가 후 로드 → 반환 view_count에 +1 반영. 태그·작성자 결합. `content`·`version` 포함.
- **create**(req) `@Transactional` → detail
  - `repo.save(Sermon.create(...))`(created_by/updated_by 자동) → `replaceLinks(SERMON, id, req.tagIds())`.
- **update**(id, req) `@Transactional` — PUT 전체 교체
  - load(404) → **명시적 version 비교**(`req.version != s.version` → `OPTIMISTIC_LOCK_CONFLICT`) → `s.update(...)` → `replaceLinks(SERMON, id, tagIds)`(null이면 전체 해제).
- **patch**(id, req) `@Transactional` — 부분 갱신
  - load(404) → version 비교 → `s.applyPatch(...)`(전달 필드만) → tagIds **제공 시에만** `replaceLinks`(null이면 태그 유지).
- **delete**(id) `@Transactional`
  - load(404) → `s.softDelete()` → `ContentTagService.cleanUp(SERMON, id)`.

> 낙관락 이중 안전망: 서비스의 명시적 version 비교가 "stale 클라이언트 버전"을 결정적으로 409로 만든다.
> load~flush 사이 동시 커밋은 JPA `@Version`이 `ObjectOptimisticLockingFailureException`을 던지고,
> 이는 `GlobalExceptionHandler`가 **이미** `OPTIMISTIC_LOCK_CONFLICT`로 매핑한다(추가 작업 불필요).

## 7. API 계약

경로 인가는 기존 `SecurityConfig`로 충족(변경 불필요): `anyRequest().permitAll()`로 `GET /api/sermons/**` 공개,
`/api/admin/**`는 `authenticated()` + 메서드 `@PreAuthorize`. `SERMON_WRITE` 권한은 V2에 이미 시드됨.

### 7.1 `SermonController` (공개)

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/api/sermons` | 목록. `?page&size&sort=preachedAt,desc` + `preacher`·`series`·`from`·`to`·`tagId`·`q`. `Page<SermonCardResponse>`(content 제외). |
| GET | `/api/sermons/{id}` | 상세. 조회수 +1. `SermonDetailResponse`. |

### 7.2 `AdminSermonController` (`@PreAuthorize("hasAuthority('SERMON_WRITE')")`)

| 메서드 | 경로 | 응답 |
|---|---|---|
| POST | `/api/admin/sermons` | 201 + detail |
| PUT | `/api/admin/sermons/{id}` | 200 + detail (전체 수정) |
| PATCH | `/api/admin/sermons/{id}` | 200 + detail (부분 수정) |
| DELETE | `/api/admin/sermons/{id}` | 204 |

### 7.3 DTO

- `SermonCreateRequest` — DB 컬럼 길이와 일치하는 `@Size(max=...)`를 **명시**(초과가 DB까지 새서 500 계열로 터지지 않게):
  `title`(@NotBlank, @Size(max=200)), `preacher`(@NotBlank, @Size(max=100)), `preachedAt`(@NotNull),
  `series`(@Size(max=100)), `scripture`(@Size(max=200)), `videoUrl`/`audioUrl`(@Size(max=500)),
  `content`(TEXT — 상한 없으나 스펙 §5 최소 검증으로 합리적 길이 상한 1개 부여), `tagIds`(List<Long>).
- `SermonUpdateRequest` — create 필드 + `version`(@NotNull). PUT 전체 교체.
- `SermonPatchRequest` — 전부 nullable + `version`(@NotNull). 전달된 필드만 적용.
- `SermonCardResponse` — id, title, preacher, series, scripture, preachedAt, viewCount, tags(`List<TagResponse>`), author(String). **content 없음**.
- `SermonDetailResponse` — 카드 전체 + content, videoUrl, audioUrl, createdAt, updatedAt, **version**(편집 재전송용), tags, author.

## 10. 비목표 / 범위 밖

- 멀티테넌시 컬럼(스펙 §2) — 절대 추가 안 함.
- 본문 HTML 변환·새니타이즈 — 프론트 책임. 서버는 마크다운 원본만 저장(최소 길이 검증만).
- 미디어 바이너리 저장 — sermon은 본문에 `media:{id}` 문자열만 둔다(별도 연결 테이블 없음).
- 전문 검색(PG full-text) — `content` 검색은 범위 밖(카드 필드 `LIKE`만).
