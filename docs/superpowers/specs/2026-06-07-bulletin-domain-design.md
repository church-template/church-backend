> 작성일: 2026-06-07 · 선행 의존: 미디어 라이브러리(D6)·MEMBER/RBAC·BaseEntity·AuthorDisplayService — **모두 완료·배선됨**.

## 1. 목표 & 범위

교회 홈페이지의 **날짜별 주보 PDF 전시**. 갤러리와 평행한 구조지만 두 가지가 다르다: **공개 조회**(설교·공지와 같이 누구나)이고, **PDF**(이미지가 아니라)를 다룬다. 쓰기만 `BULLETIN_WRITE`로 막는다.

지금까지 구현된 콘텐츠 도메인 중 **가장 단순**하다 — 본질적으로 **notice(공개 CRUD + 작성자 표시 + 낙관락) + 갤러리 사진(media FK 참조추적)의 합성**이며, 새로 도입되는 것은 §2의 nullable + `ON DELETE SET NULL` 정합성 처리 한 곳뿐이다.

### 핵심 결정 (브레인스토밍 확정)

- **주보 = 콘텐츠 엔티티**: `BaseEntity` 상속(soft delete · `@Version` 낙관락 · `created_by`/`updated_by`). 스펙 §5 작성자 표시 정책 목록(설교·공지·갤러리·**주보**)에 포함되므로 `updated_by` 기준 작성자 표시 + 탈퇴 마스킹("(탈퇴한 사용자)")을 적용한다. `version`은 표(§5.13)엔 안 그려졌지만 §5 L217 낙관락 대상에 "주보" 명시 → BaseEntity가 제공.
- **태그·조회수·본문(content) 없음**: 스펙 §5.13 표와 §5.11 `content_tags.resource_type` 모두 주보를 포함하지 않는다. `ContentResourceType`에 값 추가하지 않는다. GET 단건도 조회수 증가 없음(공지·설교와 다른 점).
- **PDF 첨부/교체 = 갤러리식 multipart 혼합**: 생성·수정 모두 `file`(신규 업로드 → media 자동 생성) **XOR** `mediaId`(기존 라이브러리 PDF 선택)를 한 요청에서 받는다. 갤러리 `addPhotos`와 동일 발상.
  - POST: `file` XOR `mediaId` **정확히 하나 필수**(주보는 PDF가 본체).
  - PATCH: `file`/`mediaId` **최대 하나**(생략 시 PDF 유지, 메타만 수정). PDF 교체 지원.
- **제목(title) 필수 입력**: 관리자 자유 입력(예: "2026 부활절 주보"). 자동 생성 없음.
- **단건 PDF 접근 정보 = `mediaId`만**: 프론트가 `GET /api/media/{mediaId}`(공개 서빙)로 열람/다운로드. `media:{id}` 철학·갤러리 `GalleryPhotoResponse`와 일관. 추가 메타(filename/size) 동봉하지 않음.
- **미디어 차단 삭제에 FK 경로로 합류**: `bulletins.media_id`(FK `=`). `BulletinReferenceProvider`는 `GalleryPhotoReferenceProvider`와 동일한 FK 기반 Provider이며, 참조 판정은 **활성 주보만**(`deleted_at IS NULL`, 스펙 §5.10 L464 그대로).

### 이미 배선되어 있어 추가 작업이 없는 부분

- `BULLETIN_WRITE` 권한 시드(`V2`), SUPER_ADMIN 전체 권한 매핑.
- `SecurityConfig`: 공개 조회 `/api/bulletins/**`는 `anyRequest permitAll`로 통과(notice·sermon과 동일), 쓰기 `/api/admin/**`는 인증 + 메서드 `@PreAuthorize`. **변경 없음.**
- 미디어 공개 서빙 `GET /api/media/{id}`, 차단형 삭제 SPI(`MediaReferenceProvider`), `MimeTypeValidator`의 PDF 매직바이트 인식(`%PDF`).

→ 본 작업은 `domain/bulletin` 패키지 신규 추가 + `MediaService` PDF 메서드 2개 추가로 **순수 가산적(additive)** 이다.

## 2. 데이터 모델 — `V12__create_bulletins.sql`

V11(gallery)이 마지막 → 주보는 **V12**. 컬럼·인덱스 관례는 V7~V11을 따른다.

### bulletins (`BaseEntity` 상속)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| id | BIGINT PK (IDENTITY) | |
| title | VARCHAR(200) NOT NULL | 예: 2026 부활절 주보 |
| service_date | DATE NOT NULL | 주보 날짜. 기본 정렬 `service_date DESC` |
| media_id | BIGINT **nullable** REFERENCES media(id) **ON DELETE SET NULL** | 주보 PDF. ⚠️ 아래 §2.1 |
| created_at / updated_at | TIMESTAMP | JPA Auditing |
| created_by / updated_by | BIGINT REFERENCES members(id) | 감사 FK (표시용 작성자 = updated_by) |
| deleted_at | TIMESTAMP | soft delete |
| version | BIGINT NOT NULL DEFAULT 0 | 낙관락 |

```sql
CREATE TABLE bulletins (
    id           BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title        VARCHAR(200) NOT NULL,
    service_date DATE         NOT NULL,
    media_id     BIGINT       REFERENCES media (id) ON DELETE SET NULL,
    created_at   TIMESTAMP    NOT NULL,
    updated_at   TIMESTAMP,
    created_by   BIGINT       REFERENCES members (id),
    updated_by   BIGINT       REFERENCES members (id),
    deleted_at   TIMESTAMP,
    version      BIGINT       NOT NULL DEFAULT 0
);

-- 목록 정렬 = service_date DESC, 미삭제만 (스펙 §6 부분 인덱스).
CREATE INDEX idx_bulletins_service_date ON bulletins (service_date DESC) WHERE deleted_at IS NULL;
-- media 차단삭제 FK 검색용 (deleted_at 조건은 Provider 쿼리가 담당 → 비부분 인덱스).
CREATE INDEX idx_bulletins_media_id ON bulletins (media_id);
```

### 2.1 ⚠️ Critical — `media_id` nullable + `ON DELETE SET NULL` (정합성 강제 결론)

갤러리와 갈리는 **유일한** 설계 지점. "선택"이 아니라 스펙 두 줄의 논리적 귀결이다:

- **스펙 §5.13 L600**: 주보는 `deleted_at` **soft delete** → 삭제해도 행이 물리적으로 남고 `media_id`를 계속 가리킨다.
- **스펙 §5.10 L464**: 참조추적 SQL이 `bulletins ... WHERE media_id = N AND deleted_at IS NULL` → **soft-deleted 주보는 미디어 차단삭제를 막지 않는다**(스펙 명시).

이 둘을 합치면: 주보 soft-delete → 그 PDF가 차단 목록에서 빠짐 → 관리자가 media 삭제(`MediaService.delete` = media 행 **hard delete**) → soft-deleted 주보 행이 여전히 그 `media_id`를 FK로 가리켜 **FK 위반**. `NOT NULL` + 실제 FK + soft delete + "삭제 허용"은 동시에 성립 불가능(죽은 행이 null이 될 수 없으므로).

**해법**: 컬럼을 nullable로 두고 `ON DELETE SET NULL`. media가 hard-delete될 때 죽은(soft-deleted) 주보 행의 `media_id`만 DB가 자동 null화 → 댕글링 0. `members.position_id ON DELETE SET NULL` 선례와 동일.

- **활성 주보는 절대 SET NULL 대상이 안 된다**: 활성 주보(`deleted_at IS NULL`)는 차단형 참조라 media 삭제가 409로 막히므로, `ON DELETE`가 발화할 일 자체가 없다.
- **"주보 1건당 PDF 1개"(L595) 불변은 앱 레이어에서 보장**: 생성 시 `file` XOR `mediaId` 필수 → 활성 주보 `media_id`는 항상 비-null. DB nullable은 오직 죽은 행 해제용.

**갤러리와 다른 이유**: `gallery_photos.media_id`는 `NOT NULL`(V11)인데, 앨범 soft-delete 시 사진 행을 **hard delete**하므로 죽은 행이 FK를 들고 남지 않는다. 주보는 본체가 soft-delete로 남아 같은 트릭을 못 쓴다.

## 4. API

### 4.1 조회 — `BulletinController` (공개, 비인증)

| 메서드 | 경로 | 설명 |
|---|---|---|
| GET | `/api/bulletins` | 목록. `@PageableDefault(size=10, sort="serviceDate", DESC)`, soft-deleted 제외 |
| GET | `/api/bulletins/{id}` | 단건(`mediaId` 포함). 조회수 증가 없음 |

### 4.2 관리 — `AdminBulletinController` (`@PreAuthorize("hasAuthority('BULLETIN_WRITE')")`, multipart)

| 메서드 | 경로 | 입력 | 응답 |
|---|---|---|---|
| POST | `/api/admin/bulletins` | `title`, `serviceDate`, `file?` XOR `mediaId?` | 201 + Detail |
| PATCH | `/api/admin/bulletins/{id}` | `version`(필수), `title?`, `serviceDate?`, `file?`/`mediaId?`(최대 하나) | 200 + Detail |
| DELETE | `/api/admin/bulletins/{id}` | — | 204 |

멀티파트라 스칼라 필드는 전부 `@RequestParam(required=false)` 자연 타입으로 받고(날짜는 `@DateTimeFormat(iso=DATE)`), `file`은 `@RequestParam MultipartFile`(갤러리/미디어 컨트롤러 관례 일치). POST/PATCH 모두 동일하며, 필수성(`title`·`serviceDate`·PATCH의 `version`)·공백·길이·XOR·non-empty file은 **서비스에서 `uploadPdf` 호출 이전에** 검증한다(§6.1 — 고아 파일 방지). 검증 실패는 모두 `INVALID_INPUT_VALUE`(400).

`required=true`를 쓰지 않는 이유: `GlobalExceptionHandler`에 `MissingServletRequestParameterException` 핸들러가 없어 `required=true` 누락은 catch-all→**500**이 된다. `required=false`+서비스 검증이면 누락도 null→서비스 400으로 일관된다. (malformed 타입 바인딩(예: `serviceDate=garbage`)은 `MethodArgumentTypeMismatchException`→500 — 기존 갤러리/미디어 컨트롤러와 동일한 기수용 동작, 후속 개선 후보.)

## 6. 서비스 로직 — `BulletinService` (`@Transactional(readOnly = true)`)

deps: `BulletinRepository`, `MediaService`, `AuthorDisplayService`. notice 서비스 답습.

- **list(pageable)**: `repository.findByDeletedAtIsNull(pageable)` → 작성자명 배치 조회(`authorDisplayService.displayNames(updatedBy 목록)`) → 카드 매핑. (필터·검색 없음 → `Specifications` 클래스 불필요.)
- **get(id)**: `findByIdAndDeletedAtIsNull` → Detail. 조회수 증가 없음.
- **create(title, serviceDate, file, mediaId, uploaderId)** `@Transactional`:
  1. **입력 검증(§6.1) — `uploadPdf` 이전에 전부 수행**.
  2. mediaId 해소: `file`이면 `mediaService.uploadPdf(file, uploaderId)`로 새 media 생성, 아니면 `mediaService.requirePdf(mediaId)`(존재+PDF 검증).
  3. `repository.save(Bulletin.create(title, serviceDate, resolvedMediaId))`.
- **patch(id, version, title, serviceDate, file, mediaId, uploaderId)** `@Transactional`:
  1. **입력 검증(§6.1) — `uploadPdf` 이전에 전부 수행** → `load(id)` → `checkVersion(version)`(불일치 시 `OPTIMISTIC_LOCK_CONFLICT`).
  2. `file`/`mediaId`가 있으면 위와 같이 새 mediaId 해소(없으면 기존 유지).
  3. `bulletin.applyPatch(title, serviceDate, newMediaIdOrNull)` (null 인자 skip — 부분 수정).
  4. `repository.flush()` → 응답 `version` 즉시 반영(stale 방지, notice 패턴).
- **delete(id)** `@Transactional`: `load(id)` → `softDelete()`. 태그 정리 없음, media 원본 보존(§2.1로 차단삭제 시 안전).

엔티티: `Bulletin.create(title, serviceDate, mediaId)` / `applyPatch(title, serviceDate, mediaId)`(각 null 인자는 기존값 유지).

### 6.1 입력 검증 순서 — 업로드보다 먼저 (Critical, 고아 파일 방지)

`MediaService.persist`(`MediaService.java:89`)는 `fileStorage.store(file)`(**비트랜잭션 디스크 I/O**)를 media row 저장보다 먼저 한다. `uploadPdf`는 `create`/`patch`의 트랜잭션에 합류(propagation REQUIRED)하므로, **`uploadPdf` 뒤에서** 주보 INSERT가 NOT NULL 위반 등으로 실패하면 트랜잭션이 롤백돼 media row는 사라지지만 **디스크 파일은 남아 레코드 없는 고아**가 된다(`persist`의 try/catch는 persist 내부 실패만 정리, 호출자 이후 실패는 못 막음).

따라서 모든 값 검증을 **`uploadPdf` 호출 이전**에 끝낸다(위반 시 `INVALID_INPUT_VALUE`, 파일 쓰기 전 fail-fast):

| 항목 | POST | PATCH |
|---|---|---|
| `title` | 비공백 + ≤200 | 제공 시 비공백 + ≤200 |
| `serviceDate` | 필수(바인딩+null 아님) | 제공 시 유효 날짜 |
| `version` | — | 필수(`checkVersion` 전 존재) |
| `file`/`mediaId` 개수 | 정확히 하나(XOR) | 최대 하나 |
| `file` 경로 | non-empty(`file.isEmpty()` 거부) | 동일 |
| `mediaId` 경로 | non-null(+`requirePdf` 존재·PDF) | 동일 |

> 검증 통과 후의 인프라 장애성 롤백(예: DB 다운)으로 인한 고아는 스펙 §5.10이 허용하는 정상 고아(관리자 차단형 수동 정리)로 간주 — 본 절은 **예측 가능한 입력 검증 실패**가 파일을 쓰지 않게 보장하는 것이 목적이다.

## 7. 미디어 도메인 변경 — `MediaService` PDF 메서드 2개 추가

현재 이미지 전용(`uploadImage`/`requireImages`)만 존재 → PDF 대칭 메서드 추가:

- `uploadPdf(MultipartFile file, Long uploaderId)`: 저장 전 매직바이트로 `application/pdf` 확정, 아니면 `INVALID_INPUT_VALUE`로 거부(고아 파일 차단). `uploadImage` 대칭.
- `requirePdf(Long mediaId)`: 기존 mediaId 존재(`RESOURCE_NOT_FOUND`) + `mime_type == application/pdf`(`INVALID_INPUT_VALUE`) 검증. `requireImages`의 단건 PDF 버전.

`MimeTypeValidator`는 이미 `%PDF` 시그니처를 인식하므로 변경 불필요.

## 8. 테스트 전략 (TDD, 80%+)

- **BulletinService**: 생성(file 경로·mediaId 경로·XOR 위반 400·비PDF 400·PDF 미존재 404) / patch(메타만·PDF 교체·version 충돌 409·404) / delete(soft delete 확인) / list(service_date DESC 정렬·페이징·작성자 표시·삭제행 제외) / get(404).
- **입력 검증이 업로드보다 먼저(§6.1, Critical)**: 검증 실패 케이스(빈 title·serviceDate 누락·XOR 위반·빈 file)에서 `mediaService.uploadPdf`가 **호출되지 않음**을 검증(Mockito `verify(mediaService, never()).uploadPdf(...)` + `INVALID_INPUT_VALUE`) → 고아 파일이 생기지 않음을 동작으로 못박는다.
- **BulletinReferenceProvider**: 활성 주보 → 참조 1건(차단), soft-deleted 주보 → 참조 0건(비차단).
- **미디어 차단삭제 회귀**: 활성 주보 → `MediaService.delete` `MEDIA_IN_USE`(409) 차단 → 주보 soft-delete → 참조 0 → `delete` 진행. (Provider 합집합 로직 검증 — FK 캐스케이드와 독립.)
- **FK 액션 회귀(`MigrationIndexTest`, Flyway-on — 핵심 검증)**: `media_id`는 평문 `Long`이라 **JPA가 FK를 생성하지 않는다** → ON DELETE SET NULL은 **Flyway 마이그레이션에만 존재**, 일반 슬라이스(ddl-auto)에선 검증 불가. 따라서 MigrationIndexTest(이미 Flyway-on)에서 `pg_constraint.confdeltype = 'n'`(또는 `information_schema.referential_constraints.delete_rule = 'SET NULL'`)로 `bulletins.media_id` FK의 SET NULL 액션을 직접 확인한다.
- **(선택) 캐스케이드 동작 E2E**: 죽은 주보의 `media_id` null화를 동작으로 확인하려면 **Flyway-on**(FK가 존재하는 스키마)에서 실행하고, L1 캐시 stale 회피를 위해 `em.clear()` 후 재조회 또는 native query(`select media_id from bulletins where id=?`)로 검증한다(JPA 엔티티 getter는 캐스케이드를 못 봄).
- **컨트롤러 인가**: 공개 조회 200(비인증), 쓰기 권한 없음 403, `BULLETIN_WRITE` 보유 시 201/200/204.
- **MigrationIndexTest**: `idx_bulletins_service_date` 부분 인덱스(`service_date` + `deleted_at IS NULL`) 검증 1건 추가.
- 작성자 표시(탈퇴 마스킹)는 `AuthorDisplayService`가 이미 검증됨 — 통합 1건으로 확인.

## 10. 비범위 (Out of Scope)

- 태그(`content_tags`) 연결 — 스펙이 주보를 태그 대상에서 제외.
- 조회수(view_count) — 스펙 §5.13 미포함.
- `/api/main` 통합 노출 — 스펙 §5.9는 설교·공지·일정만.
- `SecurityConfig` 변경 — 공개 조회는 `permitAll`, admin은 기존 규칙으로 충족.
- RBAC 시드 — `BULLETIN_WRITE` 이미 V2 존재.
- 단건 응답의 PDF 메타(filename/size) 동봉 — `mediaId`만 내리고 프론트가 `/api/media/{id}`로 처리.
- caption·정렬 등 추가 메타 — 주보는 PDF 1건이라 해당 없음.
