# 교구/부서(Department) 도메인 설계 — 이슈 #15 (D10)

스펙 §5.8 구현. 교구·부서 소개와 **자기참조 계층 구조**를 등록·조회·관리하는 단일 교회용 도메인을 추가한다.
공개 목록은 `sort_order ASC` 평배열로 내리고 프론트가 `parentId`로 트리를 조립한다. 공개 상세는 소개(마크다운)·담당 교역자·활동을 반환한다.
본문(`description`)은 마크다운 원본을 저장하고 렌더링은 프론트 책임이다(스펙 §5 본문 저장 정책).

## 0. 배경 — Event(D9)/Notice(D8) 패턴의 답습

Department는 Event·Notice가 세운 콘텐츠 도메인 패턴을 따르되, **자기참조 계층(parent_id)** 이라는 고유점이 하나 있는 형제 도메인이다.
새로운 횡단 아키텍처 결정은 없고, 기존 자산을 그대로 재사용한다.

기존 자산 재사용(신규 횡단 컴포넌트 0개):
- `MediaReferenceProvider`(media 도메인 SPI) — 본문 `media:{id}` 참조 추적. sermon·notice·event에 이은 네 번째 구현체.
- `BaseEntity`(global/common) — 감사/소프트삭제/낙관락 컬럼 상속.
- `ErrorCode`(global/exception) — 차단형 409는 `MEDIA_IN_USE`·`ROLE_IN_USE` 선례를 따라 `DEPARTMENT_HAS_CHILDREN` 1종만 신설.

Event 대비 **재사용하지 않는** 것:
- **`ContentTagService` 미사용** — 부서는 태그 대상이 아니다(스펙 §5.11 태그 풀은 sermon·notice·event·gallery_album만, `ContentResourceType`에 DEPARTMENT 없음).
- **`AuthorDisplayService` 미사용** — 부서는 작성자를 노출하지 않는다(§1 결정, Event와 동일).
- **페이지네이션 미적용** — 부서는 positions·tags 같은 **구조적 소량 데이터**라 `Page` 봉투가 아니라 평배열을 내린다(§1 결정).

## 1. 확정된 설계 결정 (브레인스토밍 합의)

| 항목 | 결정 | 근거 |
|---|---|---|
| **목록 응답 형태** | **평면 배열 + `parentId`** — `List<DepartmentCardResponse>`, `sort_order ASC`, **비페이징**. 프론트가 `parentId`로 트리 조립 | positions·tags 선례(구조적 소량 데이터는 평배열, `Page` 봉투 아님)와 일치. 백엔드 최단순(쿼리 1번), 트리/평면 양쪽 렌더 가능. 트리 조립을 백엔드 재귀 DTO로 떠안지 않는다. |
| **자식 있는 삭제** | **차단(409 `DEPARTMENT_HAS_CHILDREN`)** — 살아있는 자식이 하나라도 있으면 거부 | media `MEDIA_IN_USE`·role `ROLE_IN_USE` 차단형 삭제 철학과 일관. 데이터 손실·암묵적 트리 재구성 없음. 구현 최단순(자식 존재 여부만 확인). |
| **계층 깊이** | **무제한 + 순환만 차단** — 깊이 상한 없음, `parent_id`만 검증 | 스펙 §5.8이 깊이를 명시하지 않음 → YAGNI로 추가 제약 없이 순환 검증만. "2단계 고정" 같은 추가 규칙은 두지 않는다. |
| **순환 방지** | newParent의 `parent_id` 조상 체인을 루트까지 상향 탐색하다 `self.id`를 만나면 거부 | 트리라 부모는 하나뿐 → 후손 집합 계산보다 단순(O(depth)). 자기참조(`parentId == self`)는 별도 선검사. |
| **parent 검증 실패 매핑** | **전부 `400 INVALID_INPUT_VALUE`**, `detail`로만 구분 | 본문(body)의 참조 오류는 400이라는 코드베이스 관례(`ContentTagService` "존재하지 않는 태그…", `RoleService` "존재하지 않는 권한…"). 404는 **경로(URL)의 `{id}`** 전용으로 유지해 클라이언트 분기 모호성 제거. |
| **작성자 표시** | **노출 안 함** — 응답 DTO에 author 필드 없음 | §5.8 부서 필드가 `created_by/updated_by`를 생략하고, §5 작성자표시 정책도 부서를 제외(설교·공지·갤러리·주보만). 엔티티는 BaseEntity 상속으로 감사·낙관락 컬럼은 보유하되 API로는 내지 않는다(Event와 동일). |
| **`leader`(담당 교역자)** | **평문 `VARCHAR(100)`** — Member FK 아님 | 스펙 §5.8 필드 표기가 `leader(담당 교역자)`로 단순 필드(parent_id만 "자기참조 FK"로 명시). 교역자 이름 텍스트로 저장. |
| **`sort_order`** | `INTEGER NOT NULL`, **create 미지정 시 `max(sort_order)+10` append**(`PositionService` 선례 일치). 목록 정렬은 **`sort_order ASC, id ASC`**(동률 결정적). update/patch는 null=기존값 유지 | 기본값 0으로 두면 누락 생성이 전부 동률(0)이 되어 `sort_order ASC` 순서가 DB 의존으로 흔들린다(리뷰 Finding 2). positions와 동일하게 max+10 append하고, 명시 동률은 `id`로 tie-break해 결정적 순서를 보장. 전역 단일 정렬값 — 같은 부모의 형제는 `sort_order`로 정렬. |
| **낙관락 범위** | `version`(@Version)은 **departments 행만** 보호. update/patch는 mutation 후 **`repository.flush()`** | sermon·notice·event 정합 답습. 부서는 태그가 없어 Event의 "tag-only 미증가" 분기가 없다 — 엔티티 필드가 바뀌면 항상 N→N+1. |
| **미디어 참조 매칭** | **경계 안전**(PG 정규식 `media:{id}($\|[^0-9])`) | 순진한 `LIKE '%media:42%'`는 `media:420`도 매칭 → 잘못된 삭제 차단. sermon·notice·event 선례 그대로. 매칭 대상 컬럼은 `description`. |
| **PATCH `parentId=null` 의미** | **미변경** — 루트로 되돌리려면 PUT 사용 | Event의 `endAt` null=미변경 관례와 동일. PATCH로는 부모 교체만 가능, 부모 해제(=루트화)는 PUT 전체 교체로. 루트화는 드문 케이스라 단순화. |

## 2. 데이터 모델

### 2.1 마이그레이션 — `V10__create_departments.sql`

> V9는 `V9__create_events.sql`이 점유 중 → department는 **V10**.

```sql
-- 교구/부서 콘텐츠(스펙 §5.8). BaseEntity 상속, 감사/소프트삭제/낙관락 컬럼은 V7~V9 관례를 따른다.
-- 본문 description은 마크다운 원본(TEXT), 본문 내 이미지는 media:{id}로 참조(스펙 §5 본문 정책).
-- parent_id는 자기참조 FK(계층). soft delete만 하므로 ON DELETE 절 불필요 — 자식 차단은 앱 레벨(살아있는 자식 검사).
CREATE TABLE departments (
    id          BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,                    -- 부서명(한글 표시)
    description TEXT,                                     -- 마크다운 원본(소개·활동), media:{id} 참조
    leader      VARCHAR(100),                             -- 담당 교역자 이름(평문, FK 아님)
    parent_id   BIGINT       REFERENCES departments (id), -- 자기참조 FK(계층, nullable=루트)
    sort_order  INTEGER      NOT NULL DEFAULT 0,          -- 표시 순서; 서비스가 미지정 시 max+10 append(DEFAULT 0은 백스톱)
    created_at  TIMESTAMP    NOT NULL,                    -- BaseEntity @CreatedDate
    updated_at  TIMESTAMP,                                -- BaseEntity @LastModifiedDate
    created_by  BIGINT       REFERENCES members (id),     -- 최초 작성자(audit, 미노출)
    updated_by  BIGINT       REFERENCES members (id),     -- 마지막 수정자(audit, 미노출)
    deleted_at  TIMESTAMP,                                -- soft delete
    version     BIGINT       NOT NULL DEFAULT 0           -- 낙관적 락
);

-- 기본 정렬 = sort_order, 미삭제만(스펙 §6 부분 인덱스).
CREATE INDEX idx_departments_sort_order ON departments (sort_order) WHERE deleted_at IS NULL;
```

감사 컬럼 타입은 V3(members)·V7~V9의 관례(`TIMESTAMP`, `BIGINT`)를 따른다. `ddl-auto: validate`이므로 엔티티 매핑과 정확히 일치해야 한다(빌드로 검증).
`parent_id`의 DB FK는 "참조 행 존재"만 보장하고 `deleted_at`은 검사하지 않으므로, **미삭제 부모 검증은 앱 레벨**(서비스)에서 추가한다. soft delete만 하므로 FK RESTRICT는 실제로 발화하지 않는다(차단은 앱이 선제 처리).

### 2.2 엔티티 — `Department extends BaseEntity`

- `@Entity @Table(name = "departments")`, `@Id @GeneratedValue(IDENTITY)`.
- 필드: name(String), description(String, TEXT), leader(String), **parentId(Long — 평문 컬럼)**, sortOrder(Integer).
- **`parent_id`는 `@ManyToOne` 자기연관이 아니라 평문 `Long parentId`** 로 매핑한다. `created_by/updated_by`가 평문 Long인 코드베이스 관례와 일관하며, 목록은 `parentId` 값만 필요하고 순환 검사는 리포지토리로 체인을 따라가므로 엔티티 내비게이션이 불필요하다(지연로딩·N+1 회피). DB FK는 마이그레이션이 보장.
- `BaseEntity`에서 createdAt/updatedAt/createdBy/updatedBy/deletedAt/version/`isDeleted()`/`softDelete()` 상속.
- 생성/수정 메서드(event 관례):
  - `static Department create(name, description, leader, parentId, sortOrder)` — 팩토리. **sortOrder는 서비스가 이미 해석한 값**(null이면 max+10 append, §5)을 받아 그대로 저장.
  - `update(name, description, leader, parentId, sortOrder)` — PUT 전체 교체(감사필드 제외). **parentId=null이면 루트로 설정**(명시적 해제), **sortOrder=null이면 기존값 유지**(positions `update` 관례).
  - `applyPatch(name, description, leader, parentId, sortOrder)` — PATCH 부분 갱신(null 인자는 미변경). **parentId=null·sortOrder=null 모두 미변경**(루트화는 PUT).
- `created_by`/`updated_by`는 `AuditorAware`(SecurityContext)가 자동 주입 — 서비스가 수동 세팅하지 않는다.

## 3. 리포지토리 — `DepartmentRepository extends JpaRepository<Department, Long>`

> Specification 미사용 — 동적 필터(range/tag)가 없어 파생 쿼리로 충분(Event와 다른 점).

- `Optional<Department> findByIdAndDeletedAtIsNull(Long id)` — 상세·수정·삭제·부모/조상 조회 공용.
- `List<Department> findByDeletedAtIsNullOrderBySortOrderAscIdAsc()` — 공개 목록(평배열, **`sort_order ASC, id ASC` 결정적 정렬**).
- `Optional<Integer> findMaxSortOrder()` — `@Query("select max(d.sortOrder) from Department d where d.deletedAt is null")`. create append 기준값(없으면 서비스가 10으로 시작).
- `boolean existsByParentIdAndDeletedAtIsNull(Long parentId)` — **삭제 차단용**(살아있는 자식 존재 여부). soft-deleted 자식은 제외돼 차단하지 않는다.
- `void lockHierarchy(long key)` — `@Query(value = "select pg_advisory_xact_lock(:k)", nativeQuery = true)`. 계층 변경 트랜잭션 직렬화용(§5.3). `_xact_` 변종이라 **트랜잭션 종료 시 자동 해제**.
- `List<DepartmentRefRow> findReferencesByMedia(String pattern)` — 네이티브
  `select id as id, name as title from departments where deleted_at is null and description ~ :pattern`.
  > **`name`을 `title`로 alias** — `DepartmentRefRow`/`ContentRef` 계약은 `(id, title)` 균일 형태라, 부서의 표시명 `name`을 `title` 자리에 매핑한다(sermon·notice·event는 실제 `title` 컬럼).

## 4. 미디어 참조 — `DepartmentReferenceProvider implements MediaReferenceProvider` (`@Component`)

```text
findReferences(long mediaId):
    pattern = "media:" + mediaId + "($|[^0-9])"      // 경계 안전 — 42가 420/421에 매칭 안 됨
    rows    = repo.findReferencesByMedia(pattern)
    return rows.map(r -> new ContentRef("department", r.id(), r.title()))   // title = 부서 name
```

- **`ContentRef.type`은 소문자 `"department"`** — 미디어 참조 API(`GET /references`, `MEDIA_IN_USE` 봉투)가 노출하는 계약 값(스펙 §5.10 UNION의 `'department'`). 매칭 대상은 `description` 컬럼.
- soft-deleted 부서는 쿼리에서 제외 → 삭제된 부서가 미디어 차단형 삭제를 막지 않는다(자기 치유).
- MediaService가 모든 `MediaReferenceProvider`를 `List`로 주입받아 합집합을 구한다 — media는 department를 모른다(content → media 단방향). **MediaService 코드 변경 0**.

## 5. 서비스 — `DepartmentService` (`@Transactional(readOnly=true)`)

의존: `DepartmentRepository` **하나뿐**. `ContentTagService`·`AuthorDisplayService` 의존 없음.

- **list**() → `List<DepartmentCardResponse>`
  - `findByDeletedAtIsNullOrderBySortOrderAscIdAsc()` → `description` 제외 카드로 매핑. **비페이징**, `sort_order ASC, id ASC`.
- **get**(id) → `DepartmentDetailResponse`
  - `findByIdAndDeletedAtIsNull(id)`(없으면 404 `RESOURCE_NOT_FOUND`) → `detail(department)`. 조회수 증가 없음(순수 read-only).
- **create**(req) `@Transactional` → detail
  - **계층 락**: `req.parentId() != null`이면 맨 먼저 `repo.lockHierarchy(KEY)`(§5.3 — 부모 검증/삭제 경합 직렬화).
  - `validateParent(null, req.parentId())` — parentId가 null이 아니면 **존재·미삭제** 검증(아니면 `INVALID_INPUT_VALUE` "존재하지 않는 상위 부서입니다"). 생성 시 self가 없어 자기참조·순환은 불가.
  - **sortOrder 해석**: `req.sortOrder()`가 null이면 `repo.findMaxSortOrder().map(m -> m + 10).orElse(10)`로 append, 아니면 그대로(positions `create` 선례).
  - `repo.save(Department.create(..., resolvedSortOrder))`(created_by/updated_by 자동) → `detail`.
- **update**(id, req) `@Transactional` — PUT 전체 교체
  - **계층 락 `repo.lockHierarchy(KEY)`**(§5.3, 맨 먼저 — PUT은 항상 parentId를 세팅하므로 reparent 가능) → load(404) → **명시적 version 비교**(`req.version != e.version` → `OPTIMISTIC_LOCK_CONFLICT`) → `validateParent(id, req.parentId())` → `e.update(...)`(parentId null이면 루트화) → **`repository.flush()`** → `detail(e)`.
- **patch**(id, req) `@Transactional` — 부분 갱신
  - **계층 락**: `req.parentId() != null`이면 맨 먼저 `repo.lockHierarchy(KEY)`(§5.3). → load(404) → version 비교 → **req.parentId()가 non-null일 때만** `validateParent(id, req.parentId())` → `e.applyPatch(...)`(전달 필드만, parentId=null은 미변경) → **`repository.flush()`** → `detail(e)`.
- **delete**(id) `@Transactional`
  - **계층 락 `repo.lockHierarchy(KEY)`**(§5.3, 맨 먼저 — 자식 존재 검사와 동시 create-child 경합 직렬화) → load(404) → `existsByParentIdAndDeletedAtIsNull(id)`이면 **`DEPARTMENT_HAS_CHILDREN`(409)** → 아니면 `e.softDelete()`. (태그 없어 content_tags 정리 없음.)

### 5.1 `validateParent(Long selfId, Long parentId)` — parent/순환 검증

`parentId`가 null이면 즉시 통과한다 — PUT은 null로 호출되어 "루트화"로 통과하고, PATCH는 parentId가 null이면 이 메서드를 **아예 호출하지 않는다**(미변경).

1. **자기참조**: `selfId != null && parentId.equals(selfId)` → `INVALID_INPUT_VALUE` "자기 자신을 상위 부서로 지정할 수 없습니다".
2. **존재·미삭제**: `parent = findByIdAndDeletedAtIsNull(parentId)`, 없으면 `INVALID_INPUT_VALUE` "존재하지 않는 상위 부서입니다".
3. **순환**: `selfId != null`이면 `parent.getParentId()`부터 `parent_id` 체인을 루트(null)까지 상향 탐색 — 도중 `selfId`를 만나면 `INVALID_INPUT_VALUE` "하위 부서를 상위 부서로 지정할 수 없습니다".
   - 체인 탐색은 `findByIdAndDeletedAtIsNull`로 한 단계씩 올라가며 진행(트리라 부모 1개, O(depth)). 데이터 무결성상 사이클은 존재하지 않지만(이 검증으로 차단), 방어적으로 방문 집합으로 무한루프를 막는다.

> 낙관락 이중 안전망: 서비스의 명시적 version 비교가 "stale 클라이언트 버전"을 결정적으로 409로 만든다.
> load~flush 사이 동시 커밋은 JPA `@Version`이 `ObjectOptimisticLockingFailureException`을 던지고,
> 이는 `GlobalExceptionHandler`가 **이미** `OPTIMISTIC_LOCK_CONFLICT`로 매핑한다(추가 작업 불필요).

### 5.2 신규 에러코드 — `ErrorCode.DEPARTMENT_HAS_CHILDREN`

```java
DEPARTMENT_HAS_CHILDREN(HttpStatus.CONFLICT, "DEPARTMENT_HAS_CHILDREN", "하위 부서가 있어 삭제할 수 없습니다"),
```

`MEDIA_IN_USE`·`ROLE_IN_USE`와 평행한 도메인 고유 409. api-conventions "코드 임의 발명 금지"는 ad hoc 남발을 막는 규칙이며, 차단형 삭제라는 신규 도메인 상황의 전용 코드 신설은 기존 선례(`ROLE_IN_USE`)를 따르는 정당한 추가다. parent 검증 3종은 기존 `INVALID_INPUT_VALUE` 재사용 — 신규 코드는 이 1종뿐.

### 5.3 계층 변경 직렬화 — PostgreSQL advisory lock (리뷰 Finding 1)

**문제(write-skew):** `@Version`은 "내가 읽은 그 행"만 보호하므로, **서로 다른 행을 수정하는** 동시 트랜잭션이 만드는 구조 위반은 못 막는다.

- **사이클**: A·B가 루트일 때 Tx1 `A.parent=B`, Tx2 `B.parent=A`가 동시에 검증하면 각자 상대를 루트로 보고 통과 → 둘 다 커밋 → `A↔B` 사이클. 두 Tx가 A행·B행(disjoint)을 수정해 `@Version` 충돌이 안 난다.
- **삭제부모-자식**: Tx1 `delete(A)`(자식 검사 통과)와 Tx2 `create(parent=A)`가 동시 실행 → "삭제된 A가 살아있는 자식을 가짐"으로 자식 차단이 무력화. 같은 write-skew.

**해결:** 계층을 바꾸는 트랜잭션이 **단일 advisory lock**(`pg_advisory_xact_lock(HIERARCHY_LOCK_KEY)`, `HIERARCHY_LOCK_KEY`는 코드 상수 `long`)을 트랜잭션 맨 앞에서 획득해 **검증→수정→flush 전체를 직렬화**한다. 락 하나가 위 **두 불변식(순환 금지 + 삭제부모-자식 금지)을 동시에** 지킨다.

- **획득 경로**: `update`·`delete`는 항상, `create`·`patch`는 `parentId != null`일 때. parent 링크를 세팅하거나(사이클) 자식 존재를 검사·변동시키는(삭제 경합) 모든 경로를 덮는다. 루트 생성·부모 무관 patch는 구조 위험이 없어 락을 잡지 않는다.
- **범위·비용**: 읽기(`list`/`get`), 다른 도메인, parent를 안 건드리는 수정에는 영향 없음 — 드문 계층 변경만 순차화된다. `_xact_` 변종이라 커밋/롤백 시 자동 해제(수동 unlock·누수 없음).
- **격리수준 무관**: 기본 READ COMMITTED에서 발생하는 anomaly를 앱 레벨 직렬화로 닫는다. DB 격리수준 상향이나 트리거 없이 해결.

> 단순 `@Version`(태그 last-writer-wins 용인)과 달리, 사이클·삭제부모는 **값 경합이 아니라 트리 구조 불변식 위반**이라 직렬화로 막는다. 코드베이스에 advisory lock은 첫 도입이므로 패턴 주석을 남긴다.

## 9. 비목표 / 범위 밖

- **트리(nested) 응답** — 백엔드는 평배열만. children[] 조립은 프론트 책임(§1 결정).
- **페이지네이션** — 부서 목록은 평배열(positions·tags 동급). 콘텐츠 도메인의 `Page` 봉투 미적용.
- **계층 깊이 제한** — 무제한(순환만 차단). "2단계 고정" 등 추가 제약 없음.
- **태그** — 부서는 비태그 대상(스펙 §5.11).
- **작성자 표시** — §1 결정대로 미노출(엔티티 감사 컬럼은 보유).
- **view_count / 조회수** — 부서는 카운팅 안 함(Redis §9 대상 아님).
- **자식 재배치/승격(orphan)·일괄 삭제(cascade)** — 삭제는 차단형만. 자식은 관리자가 먼저 이동/삭제.
- 멀티테넌시 컬럼(스펙 §2) — 절대 추가 안 함.
- 본문 HTML 변환·새니타이즈 — 프론트 책임. 서버는 마크다운 원본만 저장(최소 길이 검증만).
- 미디어 바이너리 저장 — department는 본문에 `media:{id}` 문자열만 둔다(별도 연결 테이블 없음).
